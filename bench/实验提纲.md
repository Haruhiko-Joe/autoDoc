# ACCEED 消融实验设计

## 系统概述

ACCEED 系统包含两个核心 agent：
1. **文档站生成 agent**：接收代码仓库 URL，通过 5-agent 流水线（Scaffold → Checker → Decomposer → Writer → Flow Analyzer）生成结构化文档站。后续通过 PrUpdater agent 基于 PR diff 增量更新文档。
2. **文档站消费 agent**：通过 MCP 工具（`acceed` 服务器）或 doc-drill skill 消费文档站内容，回答关于代码库的问题。

---

## 实验一：文档站结构消融（生成侧）

**目标**：评估文档站各结构组件对下游问答质量的贡献。

### 自变量：3 种文档变体

已由 `bench/scripts/generate-ablation-docs.ts` 实现，产出于 `bench/data/ablation-docs/`：

| 变体 | 代号 | 描述 |
|---|---|---|
| 完整 ACCEED | `full` | 层次图 + 6 种语义边 + 叶子 Markdown 页 + `flows.json` 交互流 |
| 去边（仅树） | `no-edges` | 保留图层次结构，但清除所有节点的 `edges` 数组并清空 `flows.json`（退化为树） |
| 扁平 Markdown | `flat-md` | 通过最小 doc-drill wrapper 暴露为单层 `FlatMarkdown` 图；所有叶子页是 siblings，不保留原始层次、edges、flows、codeScope |

#### 6 种语义边类型（`EdgeType`，定义于 `src/agents/schemas/schema.ts`）

| 边类型 | 语义 |
|---|---|
| `calls` | 函数/方法调用 |
| `depends` | 模块依赖 |
| `data-flow` | 数据流动 |
| `event` | 事件发布/订阅 |
| `extends` | 继承/实现 |
| `composes` | 组合/聚合 |

#### 结构解读

- **`full`**：完整形态——层次图中节点间有 6 种语义边，节点持有 `codeScope`、`description`、`child`（指向子图或叶子页），加上 `flows.json` 提供 3–7 条跨模块交互流。
- **`no-edges`**：保留层次（top → sub-graph → leaf page 的父子嵌套），但节点间无语义关联信息，`flows.json` 为空。消费 agent 只能按父子关系导航，无法直接获知模块间的调用、数据流等关系。
- **`flat-md`**：完全退化为一个单层 doc-drill 图——`top.json` 只包含 `FlatMarkdown`，`FlatMarkdown/FlatMarkdown.json` 下所有叶子页并列，`flows.json` 为空。当前实验一中它和其他两个变体一样由 agent 通过文档入口回答问题；后续实验二可直接把这些 Markdown 页作为 RAG baseline 语料。

### 因变量

对相同 QA 问题集，仅提供对应变体的文档工具，衡量回答质量（采分点命中率 + LLM-as-judge 评分）。

### 当前实现约束

当前只实现实验一：让 agent 分别基于 `full`、`no-edges`、`flat-md` 三种文档回答同一组 QA，并用同一套 judge 评测。为保证实验一只改变“文档结构”这一变量，三种文档都统一通过 readonly doc-drill browse script 暴露给回答 agent。RAG 与 doc-drill 的消费方式对比不属于实验一，而是实验二的内容。

## 实验二：消费方式消融（消费侧）

**目标**：比较不同文档消费方式对 agent 回答质量的影响。

### 自变量：消费方式

| 消费方式 | 描述 |
|---|---|
| **RAG + 扁平 Markdown**（baseline） | 将 `flat-md` 文档作为 RAG 语料，模拟传统文档站（如 Google 开源项目文档）的消费体验 |
| **RAG + ACCEED 文档** | 将 `full` 文档（含图结构 JSON + Markdown 页）作为 RAG 语料 |
| **doc-drill skill**（MCP 工具消费） | 通过 `acceed` MCP 服务器暴露的结构化工具（`get_top` → `get_flows` → `get_graph` → `get_page` → `search_nodes`）按需渐进式钻取，遵循 skill 模板定义的 orient → flows → locate → focus → search 工作流 |

### 预期差异

- RAG 是被动检索：由 embedding 相似度决定召回内容，可能丢失结构语境。
- doc-drill skill 是主动导航：agent 按层次结构有意识地钻取，能利用边关系和交互流追踪跨模块问题。skill 模板中内置的“渐进式钻取”策略（先 `get_top` 概览，再按需深入）应比盲检索更精准、更省 token。

## 实验三：文档站 vs 源码 vs 文档站+源码

**目标**：验证文档站的独立价值和互补价值。

### 自变量：上下文来源

| 条件 | 代号 | agent 可用的信息源 |
|---|---|---|
| 仅文档站 | `doc-only` | 只能通过 MCP 工具访问 ACCEED 文档 |
| 仅源码 | `code-only` | 只能读取源代码文件（无文档） |
| 文档站 + 源码 | `doc+code` | 同时可用 MCP 文档工具 + 源码读取 |

### 关注点

1. `doc-only` 已能给出高质量回答 → 证明文档站独立可用（跨 repo 开发场景：看不到对面业务的代码）
2. `doc-only` vs `code-only` 的 token 消耗对比 → 证明文档站比直接读源码更省 token
3. `doc+code` 相比单独条件的增益 → 衡量两者互补程度

## 评估方法

### QA 对生成（Phase 1，已实现）

由 `bench/scripts/generate-qa.ts` 实现。模型（Claude / Codex）**仅基于源代码**生成高难度问答对：

- **question**：跨模块、真实场景的复杂问题（至少涉及 2 个模块/子系统）
- **goldAnswer**：基于架构理解的标准答案（模块名、函数/类名、数据流，不含行号和代码片段）
- **scoringPoints**：原子化采分点列表，每个含 `point`（可从文档验证的事实）和 `weight`（权重）
- **category**：architecture / data-flow / lifecycle / configuration / failure-mode / integration / state-management / api-contract
- **sourceEvidence**：源码证据（仅用于验证，不参与评分）

生成机制：单 session 内通过 `continue` 保持上下文连续性，避免重复问题。每 batch 完成后立即写入部分结果。

输出：`bench/data/{project}/{runId}/qa.generated.json`

### 回答验证（Phase 2，已实现）

1. 对实验一的每个文档变体（`full` / `no-edges` / `flat-md`），让 agent 回答同一组 QA 问题
2. 记录：回答内容、消耗的 token 数、工具调用次数及类型

当前实现覆盖实验一的三文档变体验证；RAG 与 doc-drill 的消费方式对比属于实验二。

### 评判（Phase 3，已实现）

- **采分点命中**：逐一检查每个 `scoringPoint` 是否被回答覆盖，加权求和
- **LLM-as-judge**：独立 judge 模型对比 `goldAnswer` 与实际回答，给出整体评分
- **Token 效率**：相同回答质量下的 token 消耗对比

## 实验矩阵总结

| 实验 | 维度 | 条件数 | 核心问题 |
|---|---|---|---|
| 实验一 | 文档结构 | 3（full / no-edges / flat-md） | 图结构和语义边对文档质量的贡献有多大？ |
| 实验二 | 消费方式 | 3（RAG-flat / RAG-acceed / doc-drill skill） | 结构化工具消费 vs 被动 RAG 检索，哪种更好？ |
| 实验三 | 信息源 | 3（doc-only / code-only / doc+code） | 文档站能否独立替代源码阅读？两者如何互补？ |

当前执行矩阵只覆盖实验一：`full`、`no-edges`、`flat-md` 三种文档变体。实验二会另行加入 `RAG-flat`、`RAG-acceed`、`doc-drill skill` 的消费方式对比。实验三主要使用完整 ACCEED 文档条件。
