# doc-drill

You have access to structured, hierarchical documentation generated by autoDoc. The docs are organized as a tree: a top-level graph per project, sub-graphs nested inside modules, markdown leaf pages at the bottom, and interaction flows for classic end-to-end cases.

**Do not try to dump everything at once.** Drill down one layer at a time, guided by relevance to the task.

## Documentation model

```
doc/{project}/
├── top.json              # project overview + top-level modules
├── flows.json            # typical cross-module interaction flows
├── {Module}/
│   ├── {Module}.json     # sub-graph: description, codeScope, child nodes
│   ├── {Leaf}.md         # leaf page: detailed doc for one component
│   └── {SubModule}/
│       ├── {SubModule}.json
│       └── ...
```

A graph's child node either points to a deeper sub-graph (`child.type == "graph"`) or to a markdown leaf page (`child.type == "page"`). Each graph carries:
- `description` — what this module does
- `codeScope` — the exact source files/dirs this module documents
- `nodes[]` — child nodes with `name`, `description`, `codeScope`, `edges[]`, `child`

Edges between sibling nodes carry relationship types: `calls`, `depends`, `data-flow`, `event`, `extends`, `composes`.

## The browse tool

Use `scripts/browse.mjs` (located in the doc-drill skill directory) to navigate. This local benchmark version reads `config.json` from the skill directory, so `{doc-dir}` and `{project}` are already configured:

```bash
# Get project overview — see all top-level modules
node {skill-dir}/scripts/browse.mjs

# Drill into a module — see its children and relationships
node {skill-dir}/scripts/browse.mjs {ModuleName}

# Keep drilling into sub-modules
node {skill-dir}/scripts/browse.mjs {Module}/{SubModule}

# Read a leaf page when you've found the right component
node {skill-dir}/scripts/browse.mjs {Module}/{LeafName} --read

# Search: find nodes by keyword across all levels
node {skill-dir}/scripts/browse.mjs --search {keyword}

# Read classic end-to-end flows
node {skill-dir}/scripts/browse.mjs --flows
```

Replace `{skill-dir}` with the actual path to the doc-drill skill directory.

## Progressive-disclosure workflow

Follow this discipline every time you need codebase context:

### 1. Orient — top-level overview
Run browse with no module path. Read the module list, their descriptions, and edges to understand the high-level architecture. **Stop here if the task only needs architectural understanding.**

### 2. Classic cases — flows
For cross-module behavior, user journeys, request lifecycles, sync jobs, or "how does this work end to end?" questions, run browse with `--flows` before drilling. Use the participants, doc paths, and code references as navigation hints. If `flows.json` is missing, continue with top / graph / page reads and source inspection.

### 3. Locate — drill into module
Pick the 1–2 most relevant modules based on descriptions and `codeScope`. Drill into them. Read child node descriptions and edges. **Stop here if you can now identify the right component.**

### 4. Focus — read leaf page
Once you've identified the specific component, read its `.md` page with `--read`. You now have detailed implementation docs: file-by-file breakdowns, data flows, key APIs.

### 5. Search — when unsure
If you can't tell which module to drill into, use `--search` with a keyword. The results tell you the matching node's module path so you can jump straight to the right depth.

## Key principles

- **Lazy loading**: Never read leaf pages until you've confirmed they're relevant via the graph hierarchy. Each page can be 5–15 KB — loading everything wastes context
- **Flow-guided tracing**: For end-to-end behavior, start from `--flows`, then drill only into modules and pages named by the relevant case
- **Edge-guided navigation**: The `edges` array tells you which siblings interact. Tracing a data flow or call chain? Follow the edges rather than reading every child
- **codeScope is ground truth**: When mapping docs back to source files, read `codeScope`. It lists the exact files/dirs a node documents
- **Breadth before depth**: At each level, scan all node descriptions first before drilling. Don't commit to the first plausible match

## Output conventions

When citing documentation findings:
- Use the module path as the citation: `Core/SessionEngine/TurnLifecycle`
- For end-to-end behavior, mention the flow title and participant path(s) you used
- Mention source files from `codeScope` when the user needs to edit code
- If a question spans multiple modules, trace the edges between them to explain the connection
