  "content": "# TerminalQuerier\n\n## Overview & Responsibilities\n\nThe `TerminalQuerier` is a timeout-free terminal query/response protocol layer within the **TerminalIO** subsystem of the **InkEngine**. It sits between the terminal output stream (stdout) and the input parser, enabling the application to interrogate the terminal's capabilities — such as which DEC private modes are active, what keyboard protocol is in use, or what the current background color is — without relying on fragile timeouts.\n\nWithin the broader architecture, `TerminalQuerier` belongs to the **TerminalIO** module inside **InkEngine**, which itself is part of the **TerminalUI** layer. The input parser (`parse-keypress.ts`) recognizes terminal response sequences in the stdin stream and routes them to the querier's `onResponse` method, while the querier writes query escape sequences to stdout.\n\n## Key Processes\n\n### The DA1 Sentinel Protocol\n\nThe core insight behind the design: every terminal since the VT100 responds to the DA1 (Primary Device Attributes) query (`CSI c`), and terminals answer queries **in order**. This means DA1 can serve as a universal \"end-of-batch\" sentinel:\n\n1. The caller queues one or more queries via `send()`, each writing its escape sequence to stdout\n2. The caller calls `flush()`, which writes a DA1 sentinel sequence\n3. When responses arrive from stdin, `onResponse()` matches them to pending queries\n4. If the DA1 response arrives before a query's response, the terminal doesn't support that query — it's resolved as `undefined`\n5. The sentinel resolves when its DA1 response arrives, draining all unmatched queries before it\n\nThis eliminates timeouts entirely: unsupported queries are detected by the absence of a response before the sentinel.\n\n### Query Lifecycle\n\n1. **Build a query**: Call a builder function (e.g., `decrqm(2026)`) which returns a `TerminalQuery<T>` containing the escape sequence to write and a type-safe matcher predicate\n2. **Send the query**: `send()` pushes a `Pending` entry onto the internal queue and writes the escape sequence to stdout. Returns a `Promise<T | undefined>`\n3. **Flush the batch**: `flush()` pushes a sentinel entry and writes `CSI c` to stdout. Returns a `Promise<void>`\n4. **Dispatch responses**: As the input parser yields `TerminalResponse` events, `onResponse()` is called for each one. It matches responses to pending queries (FIFO, first match wins), or if it's an unmatched DA1, drains everything up to the first sentinel\n\n### Concurrent Batch Isolation\n\nThe queue interleaves queries and sentinels in send order. When DA1 arrives, `onResponse` only drains up to the **first** pending sentinel (`terminal-querier.ts:204-209`). This ensures that concurrent batches from independent callers remain isolated — a sentinel only resolves its own batch.\n\n### Response Matching Strategy (`onResponse`)\n\nThe dispatcher at `terminal-querier.ts:195-211` follows this priority:\n\n1. **Match a pending query**: Scan the queue for the first `kind: 'query'` entry whose `match()` predicate returns true. If found, splice it out and resolve its promise with the response.\n2. **DA1 sentinel firing**: If no query matched and the response is DA1, find the first `kind: 'sentinel'` entry. Drain all queue entries up to and including that sentinel — resolving queries with `undefined` (unsupported) and the sentinel with `void`.\n3. **Drop unsolicited**: If neither condition matches, the response is silently ignored.\n\nA subtle detail: if a caller explicitly sends a `da1()` query, the terminal will produce **two** DA1 responses (one for the explicit query, one for the sentinel). The first DA1 matches the explicit query entry; the second fires the sentinel. This works because query matching is checked before sentinel matching.\n\n## Function Signatures\n\n### Query Builders\n\n#### `decrqm(mode: number): TerminalQuery<DecrpmResponse>`\n\nRequests DEC private mode status. The `flush()` is what triggers sentinel-based resolution of unsupported queries.\n\n## Edge Cases & Caveats\n\n- **No timeout, no rejection**: `send()` promises never reject and never time out. If you forget to call `flush()`, pending promises will hang indefinitely.\n- **Explicit DA1 queries**: If you `send(da1())` alongside other queries, the terminal produces two DA1 responses. The first matches your explicit query; the second fires the sentinel. This is handled correctly by the match-before-sentinel priority in `onResponse`.\n- **DECXCPR vs DSR ambiguity**: The `cursorPosition()` builder uses `CSI ? 6 n` (private marker) rather than `CSI 6 n` because the plain form's response (`CSI row;col R`) is ambiguous with modified function key sequences.\n- **XTVERSION over SSH**: The `xtversion()` query survives SSH connections because it goes through the pty rather than environment variables, correctly identifying the client terminal.\n- **Unsolicited responses**: Any response that doesn't match a pending query and isn't a DA1 is silently dropped.\n- **Concurrent safety**: Multiple independent callers can have overlapping query batches in flight. Sentinel draining stops at the first sentinel boundary, keeping later batches intact."
