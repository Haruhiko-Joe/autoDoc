  "content": "# SearchRendering\n\n## Overview & Responsibilities\n\nSearchRendering is an isolated side-rendering pipeline within the InkEngine's RenderPipeline, dedicated to the search functionality. It lives in `src/ink/render-to-screen.ts` and provides three core capabilities:\n\n1. **Rendering individual messages off-screen** into standalone `Screen` buffers (for search indexing)\n2. **Scanning those buffers** for case-insensitive text matches, returning cell-precise positions\n3. **Applying \"current match\" highlight styling** to a specific match position on the live screen\n\nWithin the module hierarchy, SearchRendering sits under **TerminalUI → InkEngine → RenderPipeline**. While the main RenderPipeline renders the full DOM tree to the visible terminal, SearchRendering provides a parallel path that renders a *single React element* into a disposable buffer — enabling the search system to index message content without interfering with the primary render loop.\n\n## Key Processes\n\n### Side-Render Flow (`renderToScreen`)\n\nThis is the entry point for rendering a single message into an isolated screen buffer:\n\n1. **Lazy initialization** (first call only): creates a reusable `ink-root` DOM node, a dummy `FocusManager`, object pools (`StylePool`, `CharPool`, `HyperlinkPool`), and a React reconciler container in `LegacyRoot` mode (`src/ink/render-to-screen.ts:63-82`)\n2. **React reconciliation**: synchronously updates the container with the provided React element via `updateContainerSync` + `flushSyncWork` — no async scheduling (`src/ink/render-to-screen.ts:86-88`)\n3. **Yoga layout**: sets the root width and calculates layout, deriving the natural height from the computed geometry (`src/ink/render-to-screen.ts:92-94`)\n4. **Paint to Screen**: creates a fresh `Screen` buffer, resets or creates an `Output` instance, then calls `renderNodeToOutput` to fill the buffer. Calls `output.get()` to flush queued writes into the screen's cell arrays (`src/ink/render-to-screen.ts:99-116`)\n5. **Unmount**: renders `null` into the container to clean up the React tree, but preserves the root, container, and pools for reuse by future calls (`src/ink/render-to-screen.ts:120-123`)\n6. **Performance logging**: accumulates timing for reconcile, yoga, and paint phases; logs aggregate stats every 20 calls via `logForDebugging` (`src/ink/render-to-screen.ts:125-136`)\n\n### Search Scanning Flow (`scanPositions`)\n\nScans a `Screen` buffer for all occurrences of a query string:\n\n1. **Row iteration**: walks every row of the screen buffer (`src/ink/render-to-screen.ts:159`)\n2. **Text extraction**: for each row, builds a lowercased text string from visible cells, skipping `SpacerTail`, `SpacerHead`, and `noSelect` cells — the same skip logic used by the main `applySearchHighlight` overlay (`src/ink/render-to-screen.ts:169-186`)\n3. **Code-unit-to-cell mapping**: maintains a `codeUnitToCell` array that maps character positions in the lowercased string back to actual cell indices, correctly handling surrogate pairs (emoji) and multi-unit lowercase transformations (e.g., Turkish İ → i + U+0307) (`src/ink/render-to-screen.ts:168, 181-183`)\n4. **Non-overlapping indexOf search**: finds all matches using `indexOf` with forward advancement by the query length (`src/ink/render-to-screen.ts:188-196`)\n5. Returns an array of `MatchPosition` objects with `{ row, col, len }` relative to the buffer\n\n### Current-Match Highlighting (`applyPositionedHighlight`)\n\nApplies a distinctive \"current match\" style (yellow + bold + underline) to one specific match position on the live screen:\n\n1. Translates the message-relative position to screen coordinates using `rowOffset` (the message's current screen-top position) (`src/ink/render-to-screen.ts:221`)\n2. Clips the row against `[0, screen.height)` — returns `false` if off-screen (`src/ink/render-to-screen.ts:222`)\n3. Iterates over cells in the match span, applying `stylePool.withCurrentMatch()` to each cell's style (`src/ink/render-to-screen.ts:224-229`)\n\nThis creates a two-layer highlight system: the main search overlay (`applySearchHighlight` elsewhere) renders *all* visible matches with inverse styling (\"you could go here\"), while `applyPositionedHighlight` styles the *current* match distinctively (\"you ARE here\").\n\n## Function Signatures\n\n### `renderToScreen(el: ReactElement, width: number): { screen: Screen; height: number }`\n\nRenders a single React element into a standalone `Screen` buffer.\n\n- **el**: The React element to render (must be pre-wrapped with any required context providers by the caller)\n- **width**: The screen width in columns\n- **Returns**: The rendered `Screen` buffer and the natural `height` (in rows) computed by Yoga layout\n\n> Source: `src/ink/render-to-screen.ts:59-139`\n\n### `scanPositions(screen: Screen, query: string): MatchPosition[]`\n\nScans a screen buffer for all case-insensitive occurrences of a query string.\n\n- **screen**: A `Screen` buffer (typically from `renderToScreen`)\n- **query**: The search string (case-insensitive matching)\n- **Returns**: Array of `MatchPosition` objects, each with `{ row, col, len }` relative to the buffer origin\n\n> Source: `src/ink/render-to-screen.ts:149-201`\n\n### `applyPositionedHighlight(screen: Screen, stylePool: StylePool, positions: MatchPosition[], rowOffset: number, currentIdx: number): boolean`\n\nApplies \"current match\" styling to one match position on the live screen.\n\n- **screen**: The live screen buffer to mutate\n- **stylePool**: The `StylePool` for style transformations\n- **positions**: The array of `MatchPosition` from `scanPositions`\n- **rowOffset**: Vertical offset to translate message-relative positions to screen coordinates (the message's screen-top `lo`)\n- **currentIdx**: Index into `positions` identifying which match to highlight\n- **Returns**: `true` if the highlight was applied; `false` if the index was out of bounds or the position was off-screen\n\n> Source: `src/ink/render-to-screen.ts:212-231`\n\n## Type Definitions\n\n### `MatchPosition`\n\nRepresents the location of a search match within a rendered message's bounding box.\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `row` | `number` | Row index relative to the message top (row 0 = first row of the message) |\n| `col` | `number` | Column index (cell position) where the match starts |\n| `len` | `number` | Number of cells the match spans (equals query length for ASCII; may be larger for wide characters) |\n\n> Source: `src/ink/render-to-screen.ts:26-32`\n\n## Edge Cases & Caveats\n\n- **LegacyRoot mode**: The reconciler container uses `LegacyRoot` (not `ConcurrentRoot`) because the concurrent scheduler's `flushSyncWork` leaks a small backlog across roots (~0.0003ms/call growth). This is acceptable for on-demand single-message rendering but would be pathological for bulk rendering of thousands of messages (`src/ink/render-to-screen.ts:37, 72`).\n\n- **Caller must provide contexts**: The `el` passed to `renderToScreen` must already be wrapped in any React context providers the component needs. The function only provides a minimal `FocusManager` — it does not replicate the full `App` wrapper's context tree.\n\n- **Zero-height guard**: The screen height is clamped to `Math.max(1, height)` to avoid creating a 0-height `Screen`, which could cause issues in `createScreen` (`src/ink/render-to-screen.ts:101`).\n\n- **Unicode / wide character handling**: `scanPositions` builds a `codeUnitToCell` mapping to correctly handle surrogate pairs (emoji) and locale-dependent case folding (e.g., Turkish İ lowercases to two code units). This ensures `indexOf` positions in the lowercased string map back to the correct screen cells (`src/ink/render-to-screen.ts:164-183`).\n\n- **Sync with `applySearchHighlight`**: The cell-skip logic in `scanPositions` (skipping `SpacerTail`, `SpacerHead`, `noSelect` cells) must stay in sync with the main `applySearchHighlight` function elsewhere. A TODO notes this should eventually be extracted to a shared helper (`src/ink/render-to-screen.ts:163`).\n\n- **Pool accumulation**: The `StylePool`, `CharPool`, and `HyperlinkPool` persist across calls and accumulate interned values. This is intentional — later calls benefit from higher cache hit rates (`src/ink/render-to-screen.ts:34-36`).\n\n- **Non-overlapping matches**: `scanPositions` finds non-overlapping matches only, advancing past each match by the full query length — consistent with the highlight overlay behavior (`src/ink/render-to-screen.ts:187, 195`).\n\n- **Upstream caching recommended**: The function comment notes that callers should cache results per `(message, query, width)` tuple to avoid redundant renders (`src/ink/render-to-screen.ts:56`)."
