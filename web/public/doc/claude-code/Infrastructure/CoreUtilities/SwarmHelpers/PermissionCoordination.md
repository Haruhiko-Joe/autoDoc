# Permission Coordination

## Overview & Responsibilities

Permission Coordination handles the synchronization of tool-use permission prompts between leader and teammate agents in a multi-agent swarm. Within the **Infrastructure → CoreUtilities → SwarmHelpers** hierarchy, this module ensures that when a worker agent needs user approval for a tool action (e.g., running a Bash command or editing a file), the request is forwarded to the team leader, who surfaces it in the leader's UI for the user to approve or deny.

The module is split across two files:

- **`permissionSync.ts`** (~929 lines) — The main permission synchronization engine with two communication channels: file-based (for external/pane teammates) and mailbox-based (for in-process teammates). Also handles sandbox network-access permission forwarding.
- **`leaderPermissionBridge.ts`** (~55 lines) — A lightweight registration bridge that lets in-process teammates access the leader's React-based permission confirmation UI without creating circular dependencies.

## Key Processes

### File-Based Permission Flow (External/Pane Teammates)

This channel uses JSON files on disk under `~/.claude/teams/{teamName}/permissions/` with two subdirectories: `pending/` and `resolved/`.

1. **Worker creates request** — `createPermissionRequest()` builds a `SwarmPermissionRequest` object with tool details, worker identity, and a unique ID generated by `generateRequestId()` (`permissionSync.ts:160-162`).
2. **Worker writes to pending** — `writePermissionRequest()` acquires a directory-level file lock via `lockfile.lock()`, then writes the request as JSON to `pending/{requestId}.json` (`permissionSync.ts:215-250`).
3. **Leader polls pending** — `readPendingPermissions()` scans the `pending/` directory, parses each JSON file against the Zod schema, and returns validated requests sorted oldest-first (`permissionSync.ts:256-312`).
4. **User decides** — The leader's UI presents the permission prompt. The user approves or rejects.
5. **Leader resolves** — `resolvePermission()` reads the pending file under lock, updates it with the decision, writes to `resolved/{requestId}.json`, and deletes the pending file (`permissionSync.ts:360-443`).
6. **Worker polls for response** — `pollForResponse()` checks `resolved/{requestId}.json` and converts the resolved request into a simpler `PermissionResponse` format (`permissionSync.ts:544-564`).
7. **Worker cleans up** — `removeWorkerResponse()` / `deleteResolvedPermission()` deletes the resolved file after processing (`permissionSync.ts:607-635`).

### Mailbox-Based Permission Flow (In-Process Teammates)

This channel uses the teammate mailbox system (`writeToMailbox`) for message passing, which automatically routes to in-process or file-based delivery depending on the recipient.

1. **Worker sends request** — `sendPermissionRequestViaMailbox()` looks up the leader's name from the team file, wraps the request in a `permission_request` message, and writes it to the leader's mailbox (`permissionSync.ts:676-722`).
2. **Leader processes** — The leader detects the permission request message during mailbox polling.
3. **Leader sends response** — `sendPermissionResponseViaMailbox()` creates a `permission_response` message and writes it to the worker's mailbox (`permissionSync.ts:734-783`).

### Sandbox Permission Flow

A specialized variant for sandbox network-access requests, using the same mailbox transport:

1. **Worker requests** — `sendSandboxPermissionRequestViaMailbox()` sends a `sandbox_permission_request` message with the target `host` to the leader (`permissionSync.ts:805-869`).
2. **Leader responds** — `sendSandboxPermissionResponseViaMailbox()` sends back a `sandbox_permission_response` with an `allow` boolean (`permissionSync.ts:882-928`).

### Leader Permission Bridge (In-Process UI Access)

The bridge (`leaderPermissionBridge.ts`) solves a dependency inversion problem: in-process teammates need to show permission prompts in the leader's React UI, but the runner code cannot import React components directly.

1. **REPL registers** — At startup, the leader's REPL calls `registerLeaderToolUseConfirmQueue(setter)` and `registerLeaderSetToolPermissionContext(setter)` to expose its React state setters (`leaderPermissionBridge.ts:28-33, 42-46`).
2. **Teammate retrieves** — The in-process teammate runner calls `getLeaderToolUseConfirmQueue()` and `getLeaderSetToolPermissionContext()` to push permission prompts into the leader's confirmation dialog (`leaderPermissionBridge.ts:34-36, 48-50`).
3. **Cleanup** — `unregisterLeaderToolUseConfirmQueue()` and `unregisterLeaderSetToolPermissionContext()` clear the registrations when the session ends (`leaderPermissionBridge.ts:38-40, 52-54`).

## Function Signatures

### permissionSync.ts — File-Based API

#### `createPermissionRequest(params): SwarmPermissionRequest`
Factory that builds a request object. Pulls `teamName`, `workerId`, `workerName`, and `workerColor` from environment if not provided. Throws if team name, worker ID, or worker name are missing.

#### `writePermissionRequest(request): Promise<SwarmPermissionRequest>`
Writes a pending request file with directory-level file locking. Returns the written request.

#### `readPendingPermissions(teamName?): Promise<SwarmPermissionRequest[]>`
Reads and validates all pending requests. Returns them sorted by `createdAt` ascending. Returns `[]` on missing directory or parse failures.

#### `readResolvedPermission(requestId, teamName?): Promise<SwarmPermissionRequest | null>`
Reads a single resolved request. Returns `null` if not found or invalid.

#### `resolvePermission(requestId, resolution, teamName?): Promise<boolean>`
Atomically moves a request from `pending/` to `resolved/` with the resolution data applied. Returns `false` if the pending request is not found.

#### `pollForResponse(requestId, _agentName?, teamName?): Promise<PermissionResponse | null>`
Worker-side convenience that converts a resolved request into a simplified `PermissionResponse`.

#### `cleanupOldResolutions(teamName?, maxAgeMs?): Promise<number>`
Deletes resolved files older than `maxAgeMs` (default: 1 hour). Returns the count of cleaned files.

#### `deleteResolvedPermission(requestId, teamName?): Promise<boolean>`
Deletes a single resolved file. `removeWorkerResponse` is an alias.

### permissionSync.ts — Mailbox-Based API

#### `sendPermissionRequestViaMailbox(request): Promise<boolean>`
Sends a tool-permission request to the leader's mailbox.

#### `sendPermissionResponseViaMailbox(workerName, resolution, requestId, teamName?): Promise<boolean>`
Sends a tool-permission response to a worker's mailbox.

#### `sendSandboxPermissionRequestViaMailbox(host, requestId, teamName?): Promise<boolean>`
Sends a sandbox network-access request to the leader's mailbox.

#### `sendSandboxPermissionResponseViaMailbox(workerName, requestId, host, allow, teamName?): Promise<boolean>`
Sends a sandbox network-access response to a worker's mailbox.

### permissionSync.ts — Utility

#### `isTeamLeader(teamName?): boolean`
Returns `true` if the current agent has no agent ID or its ID is `'team-lead'`.

#### `isSwarmWorker(): boolean`
Returns `true` if the agent has both a team name and an agent ID, and is not the leader.

#### `getLeaderName(teamName?): Promise<string | null>`
Reads the team file to find the leader member's name. Falls back to `'team-lead'`.

### leaderPermissionBridge.ts

#### `registerLeaderToolUseConfirmQueue(setter): void` / `getLeaderToolUseConfirmQueue(): SetToolUseConfirmQueueFn | null` / `unregisterLeaderToolUseConfirmQueue(): void`
Register, retrieve, or clear the leader's `ToolUseConfirm[]` state updater function.

#### `registerLeaderSetToolPermissionContext(setter): void` / `getLeaderSetToolPermissionContext(): SetToolPermissionContextFn | null` / `unregisterLeaderSetToolPermissionContext(): void`
Register, retrieve, or clear the leader's `ToolPermissionContext` setter function.

## Type Definitions

### `SwarmPermissionRequest` (Zod schema, `permissionSync.ts:49-86`)

| Field | Type | Description |
|---|---|---|
| `id` | `string` | Unique request ID (`perm-{timestamp}-{random}`) |
| `workerId` | `string` | Worker's `CLAUDE_CODE_AGENT_ID` |
| `workerName` | `string` | Worker's `CLAUDE_CODE_AGENT_NAME` |
| `workerColor` | `string?` | Worker's `CLAUDE_CODE_AGENT_COLOR` |
| `teamName` | `string` | Team name for routing |
| `toolName` | `string` | Tool requiring permission (e.g., `"Bash"`, `"Edit"`) |
| `toolUseId` | `string` | Original tool use ID from worker's context |
| `description` | `string` | Human-readable description of the tool use |
| `input` | `Record<string, unknown>` | Serialized tool input |
| `permissionSuggestions` | `unknown[]` | Suggested permission rules |
| `status` | `'pending' \| 'approved' \| 'rejected'` | Current status |
| `resolvedBy` | `'worker' \| 'leader'?` | Who resolved the request |
| `resolvedAt` | `number?` | Resolution timestamp |
| `feedback` | `string?` | Rejection feedback message |
| `updatedInput` | `Record<string, unknown>?` | Modified input if changed by resolver |
| `permissionUpdates` | `unknown[]?` | "Always allow" rules applied during resolution |
| `createdAt` | `number` | Creation timestamp |

### `PermissionResolution` (`permissionSync.ts:95-106`)

Internal type used when resolving a request: `decision`, `resolvedBy`, optional `feedback`, `updatedInput`, and `permissionUpdates`.

### `PermissionResponse` (`permissionSync.ts:523-536`)

Simplified response type returned by `pollForResponse()`: `requestId`, `decision` (`'approved' | 'denied'`), `timestamp`, optional `feedback`, `updatedInput`, and `permissionUpdates`.

### `SetToolUseConfirmQueueFn` / `SetToolPermissionContextFn` (`leaderPermissionBridge.ts:16-23`)

Function types matching the React state setters that the leader REPL exposes for in-process teammates.

## Configuration & Defaults

- **File storage path**: `~/.claude/teams/{teamName}/permissions/` with `pending/` and `resolved/` subdirectories
- **Cleanup max age**: 1 hour (`3600000` ms) by default for `cleanupOldResolutions()`
- **Leader detection**: Agent ID is absent or equals `'team-lead'`
- **Lock mechanism**: Directory-level `.lock` file in the `pending/` directory using `lockfile.lock()`

## Edge Cases & Caveats

- **File locking**: Both `writePermissionRequest` and `resolvePermission` acquire a directory-level lock before writing. This prevents race conditions when multiple workers submit requests simultaneously, but the lock scope covers the entire `pending/` directory.
- **Graceful degradation**: All read operations (`readPendingPermissions`, `readResolvedPermission`, `pollForResponse`) return empty/null on `ENOENT` errors rather than throwing, allowing callers to poll safely before directories exist.
- **Schema validation**: Every file read is validated against `SwarmPermissionRequestSchema`. Invalid files are logged and skipped rather than causing failures.
- **Decision terminology mismatch**: The file-based `SwarmPermissionRequest.status` uses `'rejected'`, while the legacy `PermissionResponse.decision` uses `'denied'` — `pollForResponse()` maps between them (`permissionSync.ts:556`).
- **Self-resolution**: `resolvePermission` accepts a `resolvedBy: 'worker'` option, allowing workers to resolve their own requests (e.g., if the permission was granted locally before the leader responded).
- **Mailbox routing**: `writeToMailbox` automatically selects in-process or file-based delivery based on the recipient, so the mailbox-based functions work transparently across both teammate types.
- **Bridge cleanup**: Callers must call the `unregister*` functions when the session ends; otherwise stale function references will persist in module-level variables.
