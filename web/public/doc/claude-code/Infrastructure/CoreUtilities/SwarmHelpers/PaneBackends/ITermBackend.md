  "content": "# ITermBackend\n\n## Overview & Responsibilities\n\nThis module provides the iTerm2 native split pane backend for the swarm multi-agent system. It sits within **Infrastructure → CoreUtilities → SwarmHelpers → PaneBackends** and is one of three `PaneBackend` implementations (alongside tmux and in-process). Its two files handle distinct concerns:\n\n- **`ITermBackend.ts`** — Implements the `PaneBackend` interface using the `it2` CLI to create, command, and destroy iTerm2 split panes.\n- **`it2Setup.ts`** — Manages the `it2` CLI installation workflow (detecting Python package managers, installing, verifying the Python API connection) and persists user preferences in global config.\n\nSibling backends (tmux, in-process) are selected by the detection/registry layer; ITermBackend self-registers on import so the registry can offer it when the terminal is iTerm2 and `it2` is available.\n\n## Key Processes\n\n### Pane Creation & Layout Strategy\n\nThe core logic lives in `createTeammatePaneInSwarmView()` (`ITermBackend.ts:114-240`). The layout rule is:\n\n1. **First teammate** — vertical split (`-v`) from the leader's session, placing the teammate to the right of the leader.\n2. **Subsequent teammates** — horizontal split (no `-v`) from the *last* teammate's session, stacking new teammates vertically in the right column.\n\nThe leader's session ID is extracted from the `ITERM_SESSION_ID` environment variable (`wXtYpZ:UUID` format, UUID portion after the colon). All splits use the `-s <session-id>` flag to target a specific pane, ensuring correctness even if the user clicks a different pane.\n\n### Serialized Pane Creation (Lock)\n\nA promise-chain lock (`acquirePaneCreationLock`, `ITermBackend.ts:21-31`) serializes all `createTeammatePaneInSwarmView` calls. Each caller awaits the previous lock promise and receives a `release` function to call when done. This prevents race conditions when multiple teammates are spawned concurrently — without it, two parallel splits could both read the same \"last teammate\" and produce an incorrect layout.\n\n### At-Fault Recovery\n\nWhen a split targeting a teammate session fails (`splitResult.code !== 0`), the backend doesn't immediately throw. Instead (`ITermBackend.ts:179-207`):\n\n1. It runs `it2 session list` to check whether the targeted session still exists.\n2. If the session is **confirmed dead** (not in the list), it prunes that ID from `teammateSessionIds` and retries the loop — the next iteration will target the new last teammate, or fall back to the leader if all teammates are gone.\n3. If the session is **still alive** (or the list command itself fails), it throws to avoid corrupting state on systemic failures (Python API disabled, `it2` removed, etc.).\n\nThis is bounded at O(N+1) iterations since each retry removes one session ID.\n\n### it2 CLI Setup Workflow\n\nThe setup flow in `it2Setup.ts` proceeds as:\n\n1. **Detect package manager** (`detectPythonPackageManager`, line 40-72) — probes for `uv`, `pipx`, `pip`, `pip3` in that order via `which`.\n2. **Install** (`installIt2`, line 90-144) — runs the appropriate install command (`uv tool install it2`, `pipx install it2`, or `pip install --user it2`). All installs run with `cwd: homedir()` to avoid project-level `pip.conf`/`uv.toml` that could redirect to a malicious PyPI server.\n3. **Verify** (`verifyIt2Setup`, line 152-195) — runs `it2 session list` to test the Python API connection. If this fails with API/Python-related errors, it sets `needsPythonApiEnabled: true` so the UI can prompt the user to enable iTerm2's Python API.\n4. **Persist** (`markIt2SetupComplete`, line 214-223) — writes `iterm2It2SetupComplete: true` to global config (`~/.claude.json`) so the setup prompt is not shown again.\n\n### Sending Commands to Panes\n\n`sendCommandToPane()` (`ITermBackend.ts:245-264`) delegates to `it2 session run -s <paneId> <command>`. The `-s` flag ensures the command reaches the correct pane regardless of which pane the user has focused.\n\n### Killing Panes\n\n`killPane()` (`ITermBackend.ts:320-339`) runs `it2 session close -f -s <paneId>`. The `-f` (force) flag is required because iTerm2's \"Confirm before closing\" preference would otherwise block or show a dialog when the shell process is still running. After the close attempt, the session ID is unconditionally removed from the tracking array — even if the pane was already gone, removing the stale ID is the correct behavior.\n\n## Function Defaults\n\n- **`ITERM_SESSION_ID`** (env var, set by iTerm2): Format `wXtYpZ:UUID`. The UUID portion identifies the leader's session for targeted splits.\n- **`iterm2It2SetupComplete`** (global config `~/.claude.json`): When `true`, suppresses the it2 setup prompt.\n- **`preferTmuxOverIterm2`** (global config `~/.claude.json`): When `true`, the backend registry skips iTerm2 even when available, falling back to tmux.\n\n## Edge Cases & Caveats\n\n- **Performance: many methods are no-ops.** `setPaneBorderColor`, `setPaneTitle`, `enablePaneBorderStatus`, and `rebalancePanes` do nothing because each `it2` invocation spawns a Python subprocess that communicates over iTerm2's API socket — too slow for cosmetic operations.\n- **Hide/show is unsupported.** iTerm2 has no equivalent to tmux's `break-pane`/`join-pane`, so `supportsHideShow` is `false` and both methods return `false`.\n- **Home-directory sandboxing during install.** All `installIt2` calls set `cwd: homedir()` to prevent a malicious project-local `pip.conf` or `uv.toml` from redirecting package downloads to an attacker-controlled PyPI mirror.\n- **Force-close required.** `killPane` passes `-f` to `it2 session close` because iTerm2's \"Confirm before closing\" preference would otherwise block programmatic close when a shell is running.\n- **Self-registration side effect.** The module's last line (`registerITermBackend(ITermBackend)`, line 370) is an intentional top-level side effect — importing the module registers the backend with the shared registry, avoiding circular dependency issues.\n- **Session ID validity.** The UUID returned by `it2 session split` is only valid when splitting from a specific session via `-s`. Splitting from the \"active\" session without `-s` may produce a UUID that's inaccessible if the split occurred in a different window."
