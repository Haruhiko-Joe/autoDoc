  "content": "# SessionEnvironment\n\n## Overview & Responsibilities\n\nThe `sessionEnvironment` module (`src/utils/sessionEnvironment.ts`) builds and caches a composite shell script that configures the environment for shell commands executed during a Claude Code session. It sits within **Infrastructure → CoreUtilities → SessionManagement** and is consumed by the shell execution layer to inject environment setup (e.g., virtualenv activation, conda environments) before every spawned command.\n\nThe module aggregates shell scripts from two sources:\n1. **`CLAUDE_ENV_FILE`** — an environment variable pointing to a script file provided by the parent process (e.g., an HFI trajectory runner that needs venv/conda activation to persist).\n2. **Hook-generated env files** — `.sh` files written into a per-session directory by lifecycle hooks (`Setup`, `SessionStart`, `CwdChanged`, `FileChanged`).\n\nAll scripts are concatenated into a single string, cached in memory, and served to callers until explicitly invalidated. Windows is not yet supported.\n\n## Key Processes\n\n### Script Assembly Flow\n\n1. `getSessionEnvironmentScript()` is called by the shell execution layer.\n2. If a cached value exists (`string` or `null`), it is returned immediately — no disk I/O.\n3. On a cache miss (`undefined`):\n   - If the platform is Windows, return `null` early.\n   - Read `process.env.CLAUDE_ENV_FILE`. If set, read and trim the file contents and push them into the `scripts` array.\n   - Read the per-session env directory (`~/.claude/session-env/<sessionId>/`).\n   - Filter files matching the `HOOK_ENV_REGEX` pattern (`<hooktype>-hook-<index>.sh`).\n   - Sort files by **hook type priority** (Setup=0, SessionStart=1, CwdChanged=2, FileChanged=3), then by **hook index** within each type — guaranteeing deterministic ordering.\n   - Read each file, trim whitespace, and append non-empty content to `scripts`.\n4. If no scripts were found, cache `null`. Otherwise, join all scripts with `\\n` and cache the result.\n\n### Cache Invalidation\n\nThe module uses a three-state cache variable (`src/utils/sessionEnvironment.ts:9-13`):\n\n| Value       | Meaning                           |\n|-------------|-----------------------------------|\n| `undefined` | Not yet loaded — will read disk   |\n| `null`      | Loaded, no files found — skip I/O |\n| `string`    | Loaded and ready — use cached     |\n\n`invalidateSessionEnvCache()` resets the cache to `undefined`, forcing the next call to re-read from disk. This is called when hooks write new env files or when the working directory changes.\n\n### CWD-Scoped File Clearing\n\nWhen the working directory changes, `clearCwdEnvFiles()` truncates (writes empty string to) all `cwdchanged-hook-*.sh` and `filechanged-hook-*.sh` files in the session env directory. This ensures stale CWD-specific environment doesn't leak into commands run from a new directory. The files are truncated rather than deleted so the hook system can overwrite them with new content.\n\n## Function Signatures\n\n### `getSessionEnvironmentScript(): Promise<string | null>`\n\nMain entry point. Returns the composite shell script to source before commands, or `null` if no environment scripts exist.\n\n> Source: `src/utils/sessionEnvironment.ts:60-144`\n\n### `getSessionEnvDirPath(): Promise<string>`\n\nReturns (and ensures existence of) the per-session env directory: `~/.claude/session-env/<sessionId>/`.\n\n> Source: `src/utils/sessionEnvironment.ts:15-23`\n\n### `getHookEnvFilePath(hookEvent, hookIndex): Promise<string>`\n\nReturns the file path where a specific hook should write its environment script.\n\n- **hookEvent**: `'Setup' | 'SessionStart' | 'CwdChanged' | 'FileChanged'`\n- **hookIndex**: `number` — the hook's position in the settings array\n- **Returns**: Path like `~/.claude/session-env/<sessionId>/setup-hook-0.sh`\n\n> Source: `src/utils/sessionEnvironment.ts:25-31`\n\n### `clearCwdEnvFiles(): Promise<void>`\n\nTruncates all `cwdchanged-hook-*` and `filechanged-hook-*` files in the session env directory. Silently ignores `ENOENT` errors.\n\n> Source: `src/utils/sessionEnvironment.ts:33-53`\n\n### `invalidateSessionEnvCache(): void`\n\nResets the in-memory cache, forcing the next `getSessionEnvironmentScript()` call to re-read from disk.\n\n> Source: `src/utils/sessionEnvironment.ts:55-58`\n\n## Internal Details\n\n### Hook File Naming Convention\n\nFiles must match the regex `HOOK_ENV_REGEX` (`src/utils/sessionEnvironment.ts:152-153`):\n\n```\n/^(setup|sessionstart|cwdchanged|filechanged)-hook-(\\d+)\\.sh$/\n```\n\nExamples: `setup-hook-0.sh`, `cwdchanged-hook-3.sh`.\n\n### Sort Order (`sortHookEnvFiles`)\n\nThe `HOOK_ENV_PRIORITY` map (`src/utils/sessionEnvironment.ts:146-151`) defines the ordering:\n\n| Hook Type      | Priority |\n|----------------|----------|\n| `setup`        | 0        |\n| `sessionstart` | 1        |\n| `cwdchanged`   | 2        |\n| `filechanged`  | 3        |\n\nWithin the same hook type, files are sorted by their numeric index. This ensures Setup hooks run first (establishing base environment), followed by session-level, then CWD-level, then file-level — each layer can build on the previous.\n\n## Edge Cases & Caveats\n\n- **Windows**: Returns `null` unconditionally with a debug log. Shell environment scripts are not supported on Windows.\n- **ENOENT suppression**: All file reads silently ignore `ENOENT` errors (file/directory not found), only logging other error types. This makes the module resilient to race conditions where files are deleted between `readdir` and `readFile`.\n- **Truncation, not deletion**: `clearCwdEnvFiles()` writes empty strings rather than unlinking files. This avoids race conditions with concurrent hook execution that might recreate the file.\n- **`CLAUDE_ENV_FILE` always loads first**: The parent-process env file is always prepended before any hook files, so it forms the base environment layer.\n- **Cache semantics**: After `null` is cached (no scripts found), new scripts written to disk won't be picked up until `invalidateSessionEnvCache()` is called. Callers must invalidate after hooks write new files."
