  "content": "# SessionStartHooks\n\n## Overview & Responsibilities\n\nThe SessionStartHooks module (`src/utils/sessionStart.ts`) orchestrates the execution of **SessionStart** and **Setup** lifecycle hooks whenever a Claude Code session is created, resumed, cleared, or compacted. It sits within the **Infrastructure → CoreUtilities → SessionManagement** layer and serves as the bridge between the session lifecycle and the hook execution engine.\n\nIts sibling modules handle session storage/persistence and session state, while this module specifically owns the hook-running ceremony at session boundaries. The two main exported functions — `processSessionStartHooks` and `processSetupHooks` — load plugin hooks, execute the appropriate hook type, and collect all side-effects (messages, additional context, file watch paths, and initial user prompts) into a unified return value.\n\n## Key Processes\n\n### processSessionStartHooks Flow\n\nThis is the primary entry point, called during `startup`, `resume`, `clear`, and `compact` events.\n\n1. **Bare-mode short-circuit** — If the CLI is running in `--bare` mode, returns immediately with no hooks executed (`src/utils/sessionStart.ts:47-49`). This skips even loading plugin hooks.\n\n2. **Plugin hook loading (policy-gated)** — Checks `shouldAllowManagedHooksOnly()`. If the managed-hooks-only policy is active, plugin hooks are skipped entirely. Otherwise, calls `loadPluginHooks()` wrapped in `withDiagnosticsTiming` for observability. This function is memoized, so repeated calls are near-free (`src/utils/sessionStart.ts:56-127`).\n\n3. **Agent type resolution** — Resolves the agent type from the explicit option or falls back to `getMainThreadAgentType()` from bootstrap state (`src/utils/sessionStart.ts:131`).\n\n4. **Sequential hook execution** — Iterates over the async generator returned by `executeSessionStartHooks()`, collecting four categories of results from each hook (`src/utils/sessionStart.ts:132-156`):\n   - `message` → pushed to `hookMessages`\n   - `additionalContexts` → accumulated for later bundling\n   - `initialUserMessage` → stored in the module-level `pendingInitialUserMessage` side-channel\n   - `watchPaths` → accumulated for file watcher registration\n\n5. **File watcher registration** — If any hooks returned watch paths, calls `updateWatchPaths()` to register them (`src/utils/sessionStart.ts:158-160`).\n\n6. **Context attachment** — If any hooks produced additional context strings, bundles them into a single `HookResultMessage` via `createAttachmentMessage()` with type `hook_additional_context` (`src/utils/sessionStart.ts:163-172`).\n\n7. **Return** — Returns the collected `HookResultMessage[]` array.\n\n### processSetupHooks Flow\n\nA simpler variant that runs **Setup** lifecycle hooks, triggered by `init` or `maintenance` events (`src/utils/sessionStart.ts:177-232`). It follows the same pattern as `processSessionStartHooks` but:\n- Does **not** collect `initialUserMessage` or `watchPaths`\n- Does **not** wrap `loadPluginHooks()` in diagnostics timing\n- Uses `executeSetupHooks()` instead of `executeSessionStartHooks()`\n\n### takeInitialUserMessage Side-Channel\n\nSome hooks need to inject a first prompt into the session. Rather than changing the `Promise<HookResultMessage[]>` return type (which would ripple through five callsites in `main.tsx` and `print.ts`), the module uses a module-scoped variable as a side-channel (`src/utils/sessionStart.ts:20-32`):\n\n1. During hook execution, if a hook yields `initialUserMessage`, it is stored in `pendingInitialUserMessage`\n2. Consumers call `takeInitialUserMessage()` which returns the value **and clears it** (consume-once semantics)\n\n## Function Signatures\n\n### `processSessionStartHooks(source, options?): Promise<HookResultMessage[]>`\n\nRuns all SessionStart lifecycle hooks.\n\n- **source**: `'startup' | 'resume' | 'clear' | 'compact'` — the session event that triggered hook execution\n- **options** (optional `SessionStartHooksOptions`):\n  - `sessionId?: string` — current session identifier\n  - `agentType?: string` — override for agent type (falls back to bootstrap state)\n  - `model?: string` — current model identifier\n  - `forceSyncExecution?: boolean` — force synchronous hook execution\n- **Returns**: Array of `HookResultMessage` objects to inject into the conversation\n\n### `processSetupHooks(trigger, options?): Promise<HookResultMessage[]>`\n\nRuns all Setup lifecycle hooks.\n\n- **trigger**: `'init' | 'maintenance'` — what triggered the setup hooks\n- **options** (optional):\n  - `forceSyncExecution?: boolean` — force synchronous hook execution\n- **Returns**: Array of `HookResultMessage` objects\n\n### `takeInitialUserMessage(): string | undefined`\n\nConsumes and returns any initial user message set by a SessionStart hook. Returns `undefined` if no hook set one, or if it was already consumed.\n\n## Type Definitions\n\n### `SessionStartHooksOptions`\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `sessionId` | `string?` | Session identifier passed to hook executors |\n| `agentType` | `string?` | Agent type override; defaults to main thread agent type |\n| `model` | `string?` | Current model identifier |\n| `forceSyncExecution` | `boolean?` | Forces hooks to execute synchronously |\n\n## Error Handling\n\nPlugin hook loading failures are caught and handled gracefully — the session continues without plugin hooks while project-level hooks (from `.claude/settings.json`) still execute. The error handler in `processSessionStartHooks` provides **categorized user guidance** based on the error message content (`src/utils/sessionStart.ts:86-116`):\n\n| Error Pattern | Guidance |\n|--------------|----------|\n| `Failed to clone`, `network`, `ETIMEDOUT`, `ENOTFOUND` | Network issue — check internet connection |\n| `Permission denied`, `EACCES`, `EPERM` | Permissions issue — check `~/.claude/plugins/` permissions |\n| `Invalid`, `parse`, `JSON`, `schema` | Configuration issue — check `.claude/settings.json` |\n| Other | General guidance to fix or remove problematic plugins |\n\n`processSetupHooks` uses a simpler error handler that logs a warning without categorized guidance.\n\n## Edge Cases & Caveats\n\n- **Bare mode bypass**: Both functions return `[]` immediately in `--bare` mode, skipping even plugin hook loading. This is an optimization on top of the existing bare-mode check in `executeHooks` itself.\n- **Memoized plugin loading**: `loadPluginHooks()` is memoized, so the `await` in these functions is typically a cache hit after the initial startup load.\n- **Last-write-wins for initialUserMessage**: If multiple SessionStart hooks emit `initialUserMessage`, only the last one's value is kept — each subsequent hook overwrites `pendingInitialUserMessage`.\n- **Side-channel is consume-once**: `takeInitialUserMessage()` clears the value after reading. Calling it twice returns `undefined` the second time.\n- **No warmup logic**: The source code contains an explicit warning comment not to add any \"warmup\" logic to `processSessionStartHooks` — startup performance is critical.\n- **Plugin hooks vs. project hooks**: When plugin loading fails, only plugin-sourced hooks are lost. Project-level hooks loaded via `captureHooksConfigSnapshot` remain operational."
