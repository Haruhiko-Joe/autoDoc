  "content": "# SessionState\n\n## Overview & Responsibilities\n\nThe `sessionState` module is a lightweight state machine and notification hub that tracks the session lifecycle within Claude Code. It sits within the **Infrastructure > CoreUtilities > SessionManagement** layer and serves as a single choke point through which all session state transitions flow.\n\nIts three responsibilities are:\n\n1. **Track session state** — maintains a `SessionState` value (`idle`, `running`, or `requires_action`) that represents what the session is currently doing.\n2. **Notify listeners** — fires registered callbacks whenever the state, external metadata, or permission mode changes, so consumers (the Bridge, Terminal UI, SDK event stream) stay in sync without polling.\n3. **Push metadata to external surfaces** — mirrors `pending_action` and `task_summary` into `SessionExternalMetadata` so the CCR sidebar and push notifications can show *what* the session is blocked on, not just *that* it is blocked.\n\n## Key Processes\n\n### State Transition Flow\n\nAll state changes go through `notifySessionStateChanged()` (`src/utils/sessionState.ts:92-134`). The function:\n\n1. Updates the module-level `currentState`.\n2. Fires the registered `stateListener` callback with the new state and optional `RequiresActionDetails`.\n3. Manages the `pending_action` metadata side-channel:\n   - On `requires_action` with details → pushes `pending_action` to the metadata listener.\n   - On any other transition while a pending action exists → clears it with `null` (RFC 7396 merge-patch semantics).\n4. On `idle` → clears `task_summary` so the next turn doesn't briefly flash stale progress text.\n5. Optionally emits a `session_state_changed` event to the SDK event stream (gated by `CLAUDE_CODE_EMIT_SESSION_STATE_EVENTS`).\n\n### Listener Registration\n\nThree independent listener slots exist, each set via a dedicated setter:\n\n| Setter | Listener type | Typical consumer |\n|--------|--------------|------------------|\n| `setSessionStateChangedListener` | `(state, details?) => void` | Bridge / server session manager |\n| `setSessionMetadataChangedListener` | `(metadata) => void` | CCR external metadata PUT |\n| `setPermissionModeChangedListener` | `(mode) => void` | `print.ts` → SDK `system:status` message |\n\nEach slot holds at most one listener (latest-wins). Passing `null` unregisters.\n\n### Permission Mode Notification\n\n`notifyPermissionModeChanged()` (`src/utils/sessionState.ts:148-150`) is a single choke point for permission-mode mutations. No matter how the mode changes — Shift+Tab toggle, ExitPlanMode dialog, slash command, or bridge `set_permission_mode` — the same listener fires, guaranteeing CCR and IDE clients see mode transitions in real time.\n\n## Function Signatures\n\n### `getSessionState(): SessionState`\n\nReturns the current session state. Pure getter, no side effects.\n\n> `src/utils/sessionState.ts:88-90`\n\n### `notifySessionStateChanged(state: SessionState, details?: RequiresActionDetails): void`\n\nThe primary state transition function. Updates internal state, fires the state listener, manages `pending_action` / `task_summary` metadata, and optionally emits an SDK event.\n\n- **state** — the new lifecycle state (`idle`, `running`, or `requires_action`)\n- **details** — context for `requires_action` transitions: which tool is blocked, what it's doing, and its raw input\n\n> `src/utils/sessionState.ts:92-134`\n\n### `notifySessionMetadataChanged(metadata: SessionExternalMetadata): void`\n\nDirect metadata push for fields not tied to a state transition (e.g., `task_summary` updates from the forked summarizer, `model` changes).\n\n> `src/utils/sessionState.ts:136-140`\n\n### `notifyPermissionModeChanged(mode: PermissionMode): void`\n\nFires the permission-mode listener. Called by `onChangeAppState` whenever `toolPermissionContext.mode` changes.\n\n> `src/utils/sessionState.ts:148-150`\n\n### Listener setters\n\n- `setSessionStateChangedListener(cb | null)` — states. `requires_action` means the session is blocked waiting for user input (e.g., a permission prompt).\n\n> `src/utils/sessionState.ts:1`\n\n### `RequiresActionDetails`\n\n| Field | Type | Configuration\n\n- **`CLAUDE_CODE_EMIT_SESSION_STATE_EVENTS`** — environment variable (checked via `isEnvTruthy`). When truthy, every state transition emits a `{ type: 'system', subtype: 'session_state_changed', state }` event to the SDK event stream via `enqueueSdkEvent`. Opt-in because CCR web and mobile clients' `isWorking()` heuristics currently misinterpret the trailing `idle` event as \"Running...\".\n\n## Edge Cases & Caveats\n\n- **Single-listener slots**: Each listener type supports exactly one registered callback. If two subsystems both call `setSessionStateChangedListener`, the first is silently replaced. This is by design — the module acts as a singleton relay, not a pub/sub bus.\n- **`pending_action` lifecycle**: The `hasPendingAction` flag ensures the metadata listener receives exactly one `null` clear after a `requires_action` → non-blocked transition. Repeated non-blocked transitions do not emit redundant nulls.\n- **`task_summary` cleared on idle**: When the state transitions to `idle`, `task_summary` is explicitly nulled so the next turn doesn't briefly display the previous turn's stale progress text (`src/utils/sessionState.ts:114-116`).\n- **SDK event opt-in**: The `session_state_changed` SDK event is behind `CLAUDE_CODE_EMIT_SESSION_STATE_EVENTS` to avoid breaking CCR clients. Non-CCR consumers like scmuxd and VS Code opt in to see the authoritative idle/running signal.\n- **`post_turn_summary` is `unknown`**: Intentionally untyped to avoid leaking the `PostTurnSummaryOutput` import path into `sdk.d.ts` through `agentSdkBridge`'s re-export chain.\n- **RFC 7396 null semantics**: The `pending_action: null` clear relies on RFC 7396 JSON Merge Patch semantics — the consuming side must treat an explicit `null` value as \"delete this key.\""
