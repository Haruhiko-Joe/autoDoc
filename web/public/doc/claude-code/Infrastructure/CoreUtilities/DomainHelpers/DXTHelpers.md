  "content": "# DXT Helpers\n\n## Overview & Responsibilities\n\nThe DXT helpers module provides two core utilities for working with DXT (Desktop Extension) packages: **manifest validation** and **zip extraction**. It sits within the Infrastructure → CoreUtilities → DomainHelpers layer and is consumed by the plugin loader chain whenever Claude Code installs or processes `.dxt` extension packages.\n\nThe module is split across two files:\n- `src/utils/dxt/helpers.ts` — DXT manifest parsing and validation against the MCPB schema\n- `src/utils/dxt/zip.ts` — Secure zip extraction with safety limits and file mode preservation\n\nBoth files use **lazy imports** for their heavy dependencies (`@anthropic-ai/mcpb` and `fflate`) to avoid penalizing startup time for sessions that never touch DXT packages.\n\n## Key Processes\n\n### Manifest Validation Flow\n\n1. Raw input arrives as either binary (`Uint8Array`), text (`string`), or a pre-parsed JSON object\n2. Binary input is decoded to text via `TextDecoder` (`src/utils/dxt/helpers.ts:59`)\n3. Text input is parsed to JSON using `jsonParse()` from the slow-operations utility (`src/utils/dxt/helpers.ts:45`)\n4. The `@anthropic-ai/mcpb` package is **lazy-imported** to obtain `McpbManifestSchema` (`src/utils/dxt/helpers.ts:16`). This avoids loading ~700KB of Zod v3 bound closures at startup.\n5. The schema runs `safeParse()` on the JSON object (`src/utils/dxt/helpers.ts:17`)\n6. On failure, field errors and form errors are flattened into a single semicolon-delimited message and thrown as an `Error`\n7. On success, the validated `McpbManifest` object is returned\n\n### Extension ID Generation\n\n`generateExtensionId()` (`src/utils/dxt/helpers.ts:67-88`) produces a deterministic identifier from the manifest's author name and extension name by:\n1. Lowercasing and replacing whitespace with hyphens\n2. Stripping characters outside `[a-z0-9-_.]`\n3. Collapsing consecutive hyphens and trimming leading/trailing hyphens\n4. Joining as `{author}.{name}`, optionally prefixed with `local.unpacked` or `local.dxt`\n\nThis uses the same sanitization algorithm as the directory backend to keep IDs consistent.\n\n### Zip Extraction Flow\n\n1. `readAndUnzipFile()` reads raw bytes from disk using the injectable FS implementation (`src/utils/dxt/zip.ts:212`)\n2. `unzipFile()` lazy-imports `fflate` to avoid its ~196KB of lookup tables at startup (`src/utils/dxt/zip.ts:116`)\n3. `fflate.unzipSync()` is used (instead of async) to avoid worker termination crashes in Bun (`src/utils/dxt/zip.ts:126`)\n4. Each entry passes through a **filter callback** that calls `validateZipFile()`, which checks:\n   - **File count** — max 100,000 entries\n   - **Path safety** — rejects path traversal (`../`) and absolute paths\n   - **Individual file size** — max 512 MB per file\n   - **Total uncompressed size** — max 1 GB aggregate\n   - **Compression ratio** — max 50:1 to detect zip bombs\n5. Any validation failure throws immediately, aborting extraction\n6. On success, returns a `Record<string, Uint8Array>` mapping file paths to contents\n\n### Unix File Mode Parsing\n\n`parseZipModes()` (`src/utils/dxt/zip.ts:160-203`) recovers Unix permission bits that `fflate` discards:\n1. Scans backward from the end of the zip data to find the End of Central Directory (EOCD) signature (`0x06054b50`)\n2. Reads the entry count and central directory offset from the EOCD\n3. Walks each central directory entry (`0x02014b50` signature)\n4. For entries created on Unix hosts (`versionMadeBy` high byte === 3), extracts the `st_mode` from the upper 16 bits of `externalAttr`\n5. Returns a `name → mode` map; callers should use default permissions for missing keys\n\nThis is specifically needed for the GCS/zip installation path to preserve executable bits that the git-clone path handles natively.\n\n## Function Signatures\n\n### `validateManifest(manifestJson: unknown): Promise<McpbManifest>`\n\nValidates a pre-parsed JSON object against the MCPB manifest schema. Throws on invalid input with a formatted error message.\n\n> `src/utils/dxt/helpers.ts:13-34`\n\n### `parseAndValidateManifestFromText(manifestText: string): Promise<McpbManifest>`\n\nParses a JSON string and validates it as a DXT manifest. Throws with a descriptive error if JSON parsing fails or the schema is invalid.\n\n> `src/utils/dxt/helpers.ts:39-51`\n\n### `parseAndValidateManifestFromBytes(manifestData: Uint8Array): Promise<McpbManifest>`\n\nDecodes binary data to UTF-8 text and delegates to `parseAndValidateManifestFromText()`.\n\n> `src/utils/dxt/helpers.ts:56-61`\n\n### `generateExtensionId(manifest: McpbManifest, prefix?: 'local.unpacked' | 'local.dxt'): string`\n\nProduces a deterministic extension identifier from author and extension names. The optional `prefix` scopes the ID for local unpacked or local DXT installations.\n\n> `src/utils/dxt/helpers.ts:67-88`\n\n### `isPathSafe(filePath: string): boolean`\n\nChecks a file path for path traversal patterns and absolute paths. Returns `false` if the path could escape the extraction directory.\n\n> `src/utils/dxt/zip.ts:44-58`\n\n### `validateZipFile(file: ZipFileMetadata, state: ZipValidationState): FileValidationResult`\n\nValidates a single zip entry against all safety limits, mutating the shared `state` to track running totals. Returns `{ isValid: true }` or `{ isValid: false, error: string }`.\n\n> `src/utils/dxt/zip.ts:63-102`\n\n### `unzipFile(zipData: Buffer): Promise<Record<string, Uint8Array>>`\n\nExtracts a zip archive from an in-memory buffer with full safety validation. Uses `fflate.unzipSync` internally.\n\n> `src/utils/dxt/zip.ts:113-141`\n\n### `parseZipModes(data: Uint8Array): Record<string, number>`\n\nParses Unix file modes from the zip's central directory. Returns a filename-to-mode mapping for entries from Unix hosts. Does not handle ZIP64.\n\n> `src/utils/dxt/zip.ts:160-203`\n\n### `readAndUnzipFile(filePath: string): Promise<Record<string, Uint8Array>>`\n\nReads a zip file from disk and extracts it. Re-throws `ENOENT` errors directly; wraps all other errors with context.\n\n> `src/utils/dxt/zip.ts:209-226`\n\n## Type Definitions\n\n### `ZipValidationState`\n\nInternal state tracker accumulated across all entries during extraction.\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `fileCount` | `number` | Running count of entries processed |\n| `totalUncompressedSize` | `number` | Cumulative uncompressed bytes |\n| `compressedSize` | `number` | Total compressed size of the archive |\n| `errors` | `string[]` | Collected error messages |\n\n### `ZipFileMetadata`\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `name` | `string` | Entry path within the archive |\n| `originalSize` | `number \\| undefined` | Uncompressed size if known |\n\n### `FileValidationResult`\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `isValid` | `boolean` | Whether the entry passed validation |\n| `error` | `string \\| undefined` | Reason for rejection |\n\n## Configuration & Defaults\n\nThe `LIMITS` constant (`src/utils/dxt/zip.ts:7-13`) defines all extraction safety thresholds:\n\n| Limit | Value | Purpose |\n|-------|-------|---------|\n| `MAX_FILE_SIZE` | 512 MB | Maximum size of any single extracted file |\n| `MAX_TOTAL_SIZE` | 1 GB | Maximum aggregate uncompressed size |\n| `MAX_FILE_COUNT` | 100,000 | Maximum number of entries in the archive |\n| `MAX_COMPRESSION_RATIO` | 50:1 | Upper bound before flagging as a potential zip bomb |\n| `MIN_COMPRESSION_RATIO` | 0.5:1 | Lower bound (defined but not currently enforced in validation) |\n\nThese limits are hardcoded constants — there is no runtime configuration to override them.\n\n## Edge Cases & Caveats\n\n- **Lazy imports are critical to startup performance**: Both `@anthropic-ai/mcpb` (~700KB of Zod closures) and `fflate` (~196KB of lookup tables) are lazy-imported. Converting these to top-level imports would regress startup time for all sessions.\n- **`unzipSync` is used intentionally**: The synchronous variant is chosen over the async one because `fflate`'s async path uses Web Workers, which crash on termination in Bun.\n- **ZIP64 is not supported** by `parseZipModes()`: Archives larger than 4 GB or with more than 65,535 entries will return an empty modes map. This is acceptable since marketplace zips are typically ~3.5 MB.\n- **`MIN_COMPRESSION_RATIO` is defined but unused**: The 0.5:1 lower-bound ratio appears in the `LIMITS` object but is never checked in `validateZipFile()`. It may be intended for future use.\n- **ENOENT errors pass through directly** in `readAndUnzipFile()`: File-not-found errors are re-thrown as-is so callers can distinguish missing files from corrupt archives.\n- **`parseZipModes` shares the buffer** with the input `Uint8Array` via `Buffer.from(data.buffer, data.byteOffset, data.byteLength)` — no copy is made, which is efficient but means the caller must not mutate the input during parsing."
