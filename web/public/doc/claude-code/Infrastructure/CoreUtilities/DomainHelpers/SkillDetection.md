  "content": "# Skill Change Detection\n\n## Overview & Responsibilities\n\nThe skill change detector is a file-watching subsystem that monitors skill and command directories for filesystem changes, then triggers cache invalidation and dynamic reloading so that newly added, modified, or deleted skills are picked up without restarting the CLI. It lives within the **Infrastructure > CoreUtilities > DomainHelpers** layer and is consumed by the broader Skills & Plugins system.\n\nThe entire module is a single file — `src/utils/skills/skillChangeDetector.ts` — that exports a singleton-style API with `initialize`, `dispose`, `subscribe`, and `resetForTesting` methods.\n\n## Key Processes\n\n### Initialization Flow\n\n1. `initialize()` is called once during application startup (`src/utils/skills/skillChangeDetector.ts:85-141`).\n2. A one-time callback is registered via `onDynamicSkillsLoaded()` so that when dynamic skills finish loading, memoization caches are cleared and subscribers are notified.\n3. `getWatchablePaths()` resolves up to five categories of directories to watch:\n   - **User skills**: `~/.claude/skills`\n   - **User commands**: `~/.claude/commands`\n   - **Project skills**: `.claude/skills` (resolved to absolute path)\n   - **Project commands**: `.claude/commands` (resolved to absolute path)\n   - **Additional directories**: any `--add-dir` directories' `.claude/skills` subdirectories\n4. Each candidate path is stat-checked; non-existent paths are silently skipped.\n5. A **chokidar** watcher is created with the surviving paths, listening for `add`, `change`, and `unlink` events.\n6. A cleanup function is registered via `registerCleanup()` so the watcher is closed during graceful shutdown.\n\n### File Change Handling Flow\n\n1. When chokidar fires an event, `handleChange(path)` logs the event and emits a `tengu_skill_file_changed` analytics event (`src/utils/skills/skillChangeDetector.ts:237-245`).\n2. `scheduleReload(path)` adds the path to `pendingChangedPaths` and (re)starts a debounce timer (`src/utils/skills/skillChangeDetector.ts:255-278`).\n3. After the debounce window (300 ms by default) expires with no new events, the batched reload fires:\n   - Executes `ConfigChange` hooks with the query `'skills'`. If any hook blocks, the reload is aborted.\n   - Calls `clearSkillCaches()` to wipe the loaded-skill cache.\n   - Calls `clearCommandsCache()` to reset the command registry.\n   - Calls `resetSentSkillNames()` to allow re-sending skill metadata.\n   - Emits the `skillsChanged` signal to notify all subscribers.\n\n### Disposal Flow\n\n`dispose()` (`src/utils/skills/skillChangeDetector.ts:146-164`) sets the `disposed` flag, unregisters the cleanup hook, closes the chokidar watcher, cancels any pending debounce timer, and clears all pending paths and signal subscriptions.\n\n## Function Signatures\n\n### `initialize(): Promise<void>`\n\nStarts the file watcher. Idempotent — returns immediately if already initialized or disposed.\n\n### `dispose(): Promise<void>`\n\nTears down the watcher and all internal state. The returned promise resolves when chokidar finishes closing.\n\n### `subscribe: (listener: () => void) => () => void`\n\nRegisters a callback invoked whenever skills change on disk. Returns an unsubscribe function. Delegates to the internal `skillsChanged` signal.\n\n### `resetForTesting(overrides?): Promise<void>`\n\nResets all module-level state and optionally overrides timing constants. Test-only.\n\n- `overrides.stabilityThreshold` — chokidar's `awaitWriteFinish` stability threshold (default 1000 ms)\n- `overrides.pollInterval` — chokidar's `awaitWriteFinish` poll interval (default 500 ms)\n- `overrides.reloadDebounce` — debounce window for batching reload (default 300 ms)\n- `overrides.chokidarInterval` — stat-polling interval when `usePolling` is active (default 2000 ms)\n\n## Configuration & Defaults\n\n| Constant | Value | Purpose |\n|---|---|---|\n| `FILE_STABILITY_THRESHOLD_MS` | 1000 ms | Time a file must be unchanged before chokidar considers the write complete |\n| `FILE_STABILITY_POLL_INTERVAL_MS` | 500 ms | How often chokidar checks file size/mtime during write stabilization |\n| `RELOAD_DEBOUNCE_MS` | 300 ms | Window to batch rapid file events into a single reload cycle |\n| `POLLING_INTERVAL_MS` | 2000 ms | Stat-poll interval when running under Bun (where native watchers are avoided) |\n| `USE_POLLING` | `typeof Bun !== 'undefined'` | Forces stat-polling under Bun to avoid a known deadlock (see below) |\n\n## Chokidar Watcher Configuration\n\nThe watcher is created with these options (`src/utils/skills/skillChangeDetector.ts:110-131`):\n\n- **`persistent: true`** — keeps the process alive while watching.\n- **`ignoreInitial: true`** — only fires on changes after initialization, not for pre-existing files.\n- **`depth: 2`** — skills use a `skill-name/SKILL.md` directory layout; depth 2 covers this.\n- **`awaitWriteFinish`** — waits for file writes to stabilize before emitting events.\n- **`ignored`** — filters out non-file/non-directory entries (sockets, FIFOs, devices that cause `EOPNOTSUPP` on macOS) and `.git` directories.\n- **`ignorePermissionErrors: true`** — silently skips inaccessible paths.\n- **`usePolling` / `interval`** — enables stat-polling under Bun; uses native watchers elsewhere.\n- **`atomic: true`** — handles atomic save patterns (write-to-temp then rename).\n\n## Edge Cases & Caveats\n\n### Bun Deadlock Workaround\n\nBun's native `fs.watch()` has a known deadlock in its `PathWatcherManager` (referenced as `oven-sh/bun#27469` and `#26385`). Closing a watcher on the main thread while the file watcher thread delivers events can hang both threads. Chokidar with `depth: 2` on large skill trees triggers this reliably during git operations. The workaround is to use stat-polling (`usePolling: true`) under Bun, avoiding `FSWatcher` entirely (`src/utils/skills/skillChangeDetector.ts:51-62`).\n\n### Debounce Prevents Event Storm Deadlocks\n\nWithout debouncing, bulk operations (e.g., auto-update installing a new binary, or a git operation touching many skill files) would fire dozens of events, each triggering a full cache clear and reload cycle. This churn of rapid FSWatcher watch/unwatch calls can deadlock the Bun event loop. The 300 ms debounce batches all events into a single reload (`src/utils/skills/skillChangeDetector.ts:36-42`).\n\n### ConfigChange Hook Can Block Reload\n\nThe reload fires `executeConfigChangeHooks('skills', path)` before clearing caches. If any hook returns a blocking result, the entire reload is silently skipped and a debug log is emitted. This gives external integrations veto power over skill reloads.\n\n### Dynamic Skills Callback vs. File-Change Callback\n\nThe `onDynamicSkillsLoaded` callback intentionally calls `clearCommandMemoizationCaches()` (not `clearCommandsCache()`), because `clearCommandsCache()` would call `clearSkillCaches()`, wiping the dynamic skills that were just loaded (`src/utils/skills/skillChangeDetector.ts:93-96`).\n\n### Idempotent Initialize, One-Shot Dispose\n\n`initialize()` is a no-op if called more than once or after `dispose()`. Once disposed, the module cannot be re-initialized (the `disposed` flag is never reset outside of `resetForTesting`).\n\n## Key Code Snippets\n\n### Debounced Reload Logic\n\n```typescript\n// src/utils/skills/skillChangeDetector.ts:255-278\nfunction scheduleReload(changedPath: string): void {\n  pendingChangedPaths.add(changedPath)\n  if (reloadTimer) clearTimeout(reloadTimer)\n  reloadTimer = setTimeout(async () => {\n    reloadTimer = null\n    const paths = [...pendingChangedPaths]\n    pendingChangedPaths.clear()\n    const results = await executeConfigChangeHooks('skills', paths[0]!)\n    if (hasBlockingResult(results)) {\n      logForDebugging(\n        `ConfigChange hook blocked skill reload (${paths.length} paths)`,\n      )\n      return\n    }\n    clearSkillCaches()\n    clearCommandsCache()\n    resetSentSkillNames()\n    skillsChanged.emit()\n  }, testOverrides?.reloadDebounce ??
