It sits within the **Infrastructure → CoreUtilities → DomainHelpers** layer and is consumed by higher-level modules (such as the Teleport UI and task framework) that need to determine whether the current environment can support remote execution.\n\nThe module answers one central question: **\"Can this user, in this repository, launch a background remote session right now?\"** It does so by running a battery of checks — authentication status, remote environment availability, git repository state, GitHub App installation, and policy gates — and returning a structured list of any failures.\n\n## Key Processes\n\n### Eligibility Check Flow\n\nThe main entry point is `checkBackgroundRemoteSessionEligibility()` (`src/utils/background/remote/remoteSession.ts:45-98`). It orchestrates the full precondition pipeline:\n\n1. **Policy gate** — Checks `isPolicyAllowed('allow_remote_sessions')` first. If policy blocks remote sessions, it short-circuits immediately with `policy_blocked` and skips all other checks.\n\n2. **Parallel precondition checks** — Three independent checks run concurrently via `Promise.all`:\n   - `checkNeedsClaudeAiLogin()` — Does the user need to (re-)authenticate with Claude.ai?\n   - `checkHasRemoteEnvironment()` — Does the user have at least one remote environment available?\n   - `detectCurrentRepositoryWithHost()` — Resolve the current git remote (owner, name, host).\n\n3. **Authentication result** — If login is needed, pushes `not_logged_in`.\n\n4. **Environment result** — If no remote environments are found, pushes `no_remote_environment`.\n\n5. **Git repository checks** — A cascading decision tree:\n   - If the working directory isn't inside a git repo at all → `not_in_git_repo`.\n   - If the **bundle seed gate** is active (env vars `CCR_FORCE_BUNDLE` / `CCR_ENABLE_BUNDLE`, or the `tengu_ccr_bundle_seed_enabled` feature gate), having a `.git/` directory is sufficient — remote and GitHub App checks are skipped because the session can be seeded from a local bundle.\n   - Otherwise, if no git remote is detected → `no_git_remote`.\n   - If the remote host is `github.com`, checks whether the Claude GitHub App is installed on that specific repo → `github_app_not_installed` if not.\n\nThe function returns an empty array when all checks pass.\n\n### GitHub App Installation Check\n\n`checkGithubAppInstalled()` (`src/utils/background/remote/preconditions.ts:78-158`) verifies that the Claude GitHub App has been installed on a specific repository:\n\n1. Retrieves the OAuth access token and organization UUID.\n2. Makes an authenticated GET request to the Claude API endpoint: `/api/oauth/organizations/{orgUUID}/code/repos/{owner}/{repo}`.\n3. Inspects `response.data.status.app_installed` — if the `status` object is `null`, the app is not installed.\n4. Any 4xx HTTP error is treated as \"not installed\"; network/server errors also default to `false`.\n5. Supports an optional `AbortSignal` for cancellation and uses a 15-second timeout.\n\n### Tiered Repo Access Check\n\n`checkRepoForRemoteAccess()` (`src/utils/background/remote/preconditions.ts:221-235`) provides a two-tier fallback strategy for determining repo accessibility:\n\n1. **GitHub App** — preferred method; checked first via `checkGithubAppInstalled()`.\n2. **Token sync** — fallback, gated behind the `tengu_cobalt_lantern` feature flag; checks `checkGithubTokenSynced()` to see if the user has synced their GitHub credentials via `/web-setup`.\n3. Returns `{ hasAccess, method }` where `method` is `'github-app'`, `'token-sync'`, or `'none'`.\n\n## Function Signatures\n\n### `checkBackgroundRemoteSessionEligibility(options?): Promise<BackgroundRemoteSessionPrecondition[]>`\n\nMain eligibility checker. This is intentional — bundled sessions can be seeded from local repo content without needing GitHub access.\n- **GitHub-specific path**: The GitHub App installation check only runs when the detected remote host is exactly `github.com`. Repos hosted on other platforms (GitLab, Bitbucket, self-hosted GitHub Enterprise) skip the app check entirely after passing the remote detection step.\n- **`checkIsGitClean()` ignores untracked files**: This is deliberate — untracked files won't be lost during branch switching operations that teleport may perform.\n- **15-second timeout**: Both `checkGithubAppInstalled` and `checkGithubTokenSynced` use a 15-second HTTP timeout, which can add latency on slow connections."
