  "content": "# Post-Sampling Hooks & Skill Improvement\n\n## Overview & Responsibilities\n\nThis module provides the **post-sampling hook infrastructure** and its primary consumer: **automatic skill improvement detection**. It sits within the HooksEngine subsystem of DomainHelpers (under CoreUtilities → Infrastructure) and enables side-channel operations to run after the main Claude model finishes generating a response.\n\nThe module is composed of three files with distinct roles:\n\n- **`postSamplingHooks.ts`** — A minimal internal registry for callbacks executed after model sampling completes.\n- **`apiQueryHookHelper.ts`** — A generic factory for creating hooks that make side-channel LLM API calls with custom message building and response parsing.\n- **`skillImprovement.ts`** — The concrete feature that uses both of the above to detect user preferences/corrections during skill execution and automatically rewrite skill files.\n\n## Key Processes\n\n### Post-Sampling Hook Lifecycle\n\n1. At initialization, hooks are registered via `registerPostSamplingHook()` into a module-scoped array (`postSamplingHooks.ts:25`)\n2. After the main model sampling completes, the query engine calls `executePostSamplingHooks()` with the full conversation context (`postSamplingHooks.ts:45-70`)\n3. Each registered hook is executed **sequentially**. Errors are caught and logged but never propagate — hooks cannot crash the main conversation loop (`postSamplingHooks.ts:62-69`)\n\n### API Query Hook Factory Flow\n\n`createApiQueryHook()` (`apiQueryHookHelper.ts:56-141`) returns a closure that, when called:\n\n1. **Gate check** — Calls `config.shouldRun()` to decide whether this hook should fire\n2. **Message construction** — Calls `config.buildMessages()` to build the prompt for the side-channel LLM call\n3. **API call** — Invokes `queryModelWithoutStreaming()` with:\n   - Thinking disabled, temperature 0 (deterministic)\n   - Optional custom system prompt or inherited from context\n   - Configurable tool inclusion (on by default, can be disabled)\n   - A fresh abort controller (independent of the main conversation)\n4. **Response parsing** — Extracts text content and delegates to `config.parseResponse()`\n5. **Result logging** — Calls `config.logResult()` with either a success or error result, including a UUID for correlation\n\n### Skill Improvement Detection\n\n`createSkillImprovementHook()` (`skillImprovement.ts:68-173`) creates a hook that periodically analyzes conversation for skill improvement suggestions:\n\n1. **Gating** (`shouldRun`, lines 75-92):\n   - Only runs in the main REPL thread (`querySource === 'repl_main_thread'`)\n   - Requires an active project skill (loaded from `projectSettings:` path)\n   - Only triggers every **5 user messages** (`TURN_BATCH_SIZE`), tracking the count with `lastAnalyzedCount`\n\n2. **Analysis prompt** (`buildMessages`, lines 94-127):\n   - Sends only messages since the last analysis (incremental window via `lastAnalyzedIndex`)\n   - Includes the current skill definition for context\n   - Asks the LLM to detect user preferences, corrections, or process additions that should be persisted\n   - Expected output: a JSON array inside `<updates>` tags\n\n3. **Response parsing** (`parseResponse`, lines 134-144):\n   - Extracts `<updates>` tag content and JSON-parses it into `SkillUpdate[]`\n   - Gracefully returns `[]` on any parse failure\n\n4. **State update** (`logResult`, lines 146-167):\n   - On non-empty results, logs an analytics event (`tengu_skill_improvement_detected`)\n   - Stores the suggestion in `appState.skillImprovement` for downstream consumption\n\n### Applying Skill Improvements\n\n`applySkillImprovement()` (`skillImprovement.ts:188-267`) is a fire-and-forget function that rewrites the skill file:\n\n1. Reads the current `SKILL.md` file from `.claude/skills/<name>/SKILL.md` relative to CWD\n2. Sends the current content plus a formatted update list to the LLM with instructions to integrate changes naturally while preserving frontmatter and style\n3. Extracts the rewritten content from `<updated_file>` tags in the response\n4. Writes the updated content back to disk\n\nBoth LLM calls (detection and application) use `getSmallFastModel()` — a lightweight model suitable for classification and simple editing tasks.\n\n## Function Signatures\n\n### `postSamplingHooks.ts`\n\n#### `registerPostSamplingHook(hook: PostSamplingHook): void`\nAdds a hook to the internal registry. Hooks are called in registration order.\n\n#### `clearPostSamplingHooks(): void`\nRemoves all registered hooks. Intended for testing.\n\n#### `executePostSamplingHooks(messages, systemPrompt, userContext, systemContext, toolUseContext, querySource?): Promise<void>`\nRuns all registered hooks sequentially with the provided REPL context. Swallows individual hook errors.\n\n### `apiQueryHookHelper.ts`\n\n#### `createApiQueryHook<TResult>(config: ApiQueryHookConfig<TResult>): (context: ApiQueryHookContext) => Promise<void>`\nFactory function that returns a hook closure. The generic `TResult` type parameterizes the parsed response shape.\n\n### `skillImprovement.ts`\n\n#### `initSkillImprovement(): void`\nBootstrap function gated behind the `SKILL_IMPROVEMENT` feature flag and the `tengu_copper_panda` GrowthBook feature. Registers the skill improvement hook into the post-sampling registry.\n\n#### `applySkillImprovement(skillName: string, updates: SkillUpdate[]): Promise<void>`\nApplies detected improvements by rewriting the skill's `SKILL.md` file via a side-channel LLM call. Fire-and-forget — errors are logged but not thrown.\n\n## Type Definitions\n\n### `REPLHookContext`\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `messages` | `Message[]` | Full conversation history including assistant responses |\n| `systemPrompt` | `SystemPrompt` | Active system prompt |\n| `userContext` | `Record<string, string>` | User-provided context key-value pairs |\n| `systemContext` | `Record<string, string>` | System-provided context key-value pairs |\n| `toolUseContext` | `ToolUseContext` | Tool execution context with app state, permissions, and options |\n| `querySource` | `QuerySource` (optional) | Origin of the query (e.g., `'repl_main_thread'`) |\n\n### `ApiQueryHookContext`\n\nExtends `REPLHookContext` with:\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `queryMessageCount` | `number` (optional) | Populated by the factory after `buildMessages` — the number of messages sent |\n\n### `ApiQueryHookConfig<TResult>`\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `name` | `QuerySource` | Identifier used for analytics and logging |\n| `shouldRun` | `(ctx) => Promise<boolean>` | Gate function — return `false` to skip |\n| `buildMessages` | `(ctx) => Message[]` | Constructs the message payload for the API call |\n| `systemPrompt` | `string` (optional) | Override the context's system prompt |\n| `useTools` | `boolean` (optional, default `true`) | Whether to include tools in the API call |\n| `parseResponse` | `(content, ctx) => TResult` | Transforms raw text response into typed result |\n| `logResult` | `(result, ctx) => void` | Handles the result (analytics, state updates) |\n| `getModel` | `(ctx) => string` | Returns the model ID; must be a function for lazy loading |\n\n### `ApiQueryResult<TResult>`\n\nDiscriminated union with `type: 'success' | 'error'`:\n- **Success**: includes `result: TResult`, `messageId`, `model`, `uuid`\n- **Error**: includes `error: Error`, `uuid`\n\n### `SkillUpdate`\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `section` | `string` | Which step/section of the skill to modify (or `'new step'`) |\n| `change` | `string` | What to add or modify |\n| `reason` | `string` | The user message that prompted this update |\n\n## Configuration & Defaults\n\n- **`TURN_BATCH_SIZE = 5`** — The skill improvement hook runs once every 5 user messages (`skillImprovement.ts:31`)\n- **Feature gates**: `initSkillImprovement()` requires both a compile-time `feature('SKILL_IMPROVEMENT')` flag and a runtime GrowthBook flag (`tengu_copper_panda`) to activate\n- **Model**: Both detection and application use `getSmallFastModel()` — a fast, low-cost model for classification tasks\n- **Temperature**: Set to `0` for deterministic output in both API calls\n- **Skill file path**: `.claude/skills/<skillName>/SKILL.md` relative to the current working directory\n\n## Edge Cases & Caveats\n\n- **Hooks are fire-and-forget**: All post-sampling hooks swallow their own errors. A failing hook never interrupts the user's conversation.\n- **Incremental analysis window**: The skill improvement hook tracks `lastAnalyzedIndex` and only sends messages since the last analysis to the LLM. This means it uses a **sliding window** rather than re-analyzing the full conversation each time.\n- **Stateful closure**: `lastAnalyzedCount` and `lastAnalyzedIndex` live in the closure created by `createSkillImprovementHook()`, so they persist across hook invocations within a session but reset if the hook is re-initialized.\n- **No tools in detection**: The skill improvement detection call sets `useTools: false` (`skillImprovement.ts:132`) — the LLM cannot invoke tools during analysis.\n- **Frontmatter preservation**: The apply step explicitly instructs the LLM to preserve the `---` frontmatter block, since that contains skill metadata.\n- **Race condition potential**: `applySkillImprovement()` reads the file, sends it to the LLM, and writes back. If the file is modified between read and write, those changes will be lost. This is acceptable because skill files rarely change mid-session.\n- **State-based handoff**: Detection stores results in `appState.skillImprovement`, but does not call `applySkillImprovement()` directly. The application step is triggered elsewhere in the system (likely the UI or command layer), creating a decoupled two-phase flow.\n- **Message truncation**: `formatRecentMessages()` truncates each message to 500 characters (`skillImprovement.ts:53`) to keep the analysis prompt compact."
