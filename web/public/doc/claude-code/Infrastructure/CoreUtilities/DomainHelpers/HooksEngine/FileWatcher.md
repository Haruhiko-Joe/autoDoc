It sits within the **HooksEngine** subsystem under **Infrastructure ‚Üí CoreUtilities ‚Üí DomainHelpers**, acting as the reactive bridge between the filesystem and the hook execution pipeline.\n\nIt uses [chokidar](https://github.com/paulmillr/chokidar) to watch specific file paths and fires two types of hooks:\n\n- **FileChanged** ‚Äî when a watched file is added, modified, or deleted\n- **CwdChanged** ‚Äî when the working directory changes mid-session\n\nThe module manages both static watch paths (derived from hook configuration matchers) and dynamic watch paths (returned by hook execution output), and delivers hook results to the UI via a pluggable notifier callback.\n\n## Key Processes\n\n### Initialization Flow\n\n1. `initializeFileChangedWatcher(cwd)` is called once with the initial working directory (`fileChangedWatcher.ts:28-46`)\n2. Reads the current hook configuration via `getHooksConfigFromSnapshot()` to check for `CwdChanged` or `FileChanged` hooks\n3. If environment hooks exist, registers a cleanup handler via `registerCleanup()` to dispose the watcher on shutdown\n4. Resolves watch paths from hook matchers and starts the chokidar watcher\n5. The `initialized` flag prevents duplicate initialization\n\n### Watch Path Resolution\n\nThe `resolveWatchPaths()` function (`fileChangedWatcher.ts:48-65`) builds the full list of paths to watch by combining two sources:\n\n- **Static paths**: Extracted from each `FileChanged` hook's `matcher` field ‚Äî a pipe-separated list of filenames (e.g., `\".envrc|.env\"`). Relative names are joined with the current working directory; absolute paths are used as-is.\n- **Dynamic paths**: Accumulated from hook execution results via `updateWatchPaths()`. These allow hooks to programmatically request watching of additional files.\n\nThe result is deduplicated using a `Set`.\n\n### File Change Handling\n\n1. Chokidar emits `change`, `add`, or `unlink` events\n2. `handleFileEvent()` (`fileChangedWatcher.ts:80-106`) delegates to `executeFileChangedHooks(path, event)`\n3. Errors during hook execution are caught, logged, and forwarded to the notifier\n\n### Working Directory Change Flow\n\n`onCwdChangedForHooks(oldCwd, newCwd)` (`fileChangedWatcher.ts:133-175`) handles directory changes:\n\n1. Skips if old and new directories are identical\n2. Re-reads hook config from the latest snapshot (picks up mid-session config changes)\n3. Returns early if no environment hooks are configured\n4. Clears cached CWD environment files via `clearCwdEnvFiles()`\n5. Executes `CwdChanged` hooks, which may return new dynamic watch paths\n6. If the paths have changed, it stores them and calls `restartWatching()`, which closes the existing watcher and creates a new one with the combined static + dynamic path list.\n\n## Function Signatures\n\n### `initializeFileChangedWatcher(cwd: string): void`\n\nOne-time initialization. These determine which files are watched.\n- **`CwdChanged` hooks**: Their presence (along with `FileChanged` hooks) determines whether the watcher activates at all.\n\nChokidar is configured with:\n\n| Option | Value | Purpose |\n|--------|-------|---------|\n| `persistent` | `true` | Keeps the process alive while watching |\n| `ignoreInitial` | `true` | Skips events for files that already exist at watch start |\n| `awaitWriteFinish.stabilityThreshold` | `500ms` | Waits for writes to stabilize before firing events |\n| `awaitWriteFinish.pollInterval` | `200ms` | Polling interval during write stabilization |\n| `ignorePermissionErrors` | `true` | Silently skips files that can't be accessed |\n\n## Edge Cases & Caveats\n\n- **Single initialization**: `initializeFileChangedWatcher` silently no-ops on subsequent calls. The module is designed for one-time setup per session.\n- **No hooks, no watcher**: If no `FileChanged` or `CwdChanged` hooks are configured, the chokidar watcher is never created, avoiding unnecessary resource usage.\n- **Fire-and-forget execution**: `handleFileEvent` calls hook execution with `void` (fire-and-forget). File change events do not block the main event loop.\n- **Dynamic path deduplication**: `updateWatchPaths` uses sorted-array comparison to avoid unnecessary watcher restarts when the same paths are reported repeatedly.\n- **CWD change replaces dynamic paths**: When the working directory changes, the entire dynamic watch path list is replaced (not merged) with the `CwdChanged` hook result. This ensures stale paths from the old directory don't persist.\n- **Mid-session config changes**: `onCwdChangedForHooks` re-reads the hook config snapshot on every CWD change, so hooks added or removed during a session are picked up.\n- **Cleanup registration**: The watcher registers a cleanup handler only if environment hooks exist, ensuring graceful shutdown closes the chokidar instance.\n- **Error resilience**: Both `handleFileEvent` and `onCwdChangedForHooks` catch errors from hook execution and forward them to the notifier rather than crashing.\n\n## Key Code Snippets\n\n### Chokidar Watcher Setup\n\n```typescript\n// src/utils/hooks/fileChangedWatcher.ts:67-78\nfunction startWatching(paths: string[]): void {\n  watcher = chokidar.watch(paths, {\n    persistent: true,\n    ignoreInitial: true,\n    awaitWriteFinish: { stabilityThreshold: 500, pollInterval: 200 },\n    ignorePermissionErrors: true,\n  })\n  watcher.on('change', p => handleFileEvent(p, 'change'))\n  watcher.on('add', p => handleFileEvent(p, 'add'))\n  watcher.on('unlink', p => handleFileEvent(p, 'unlink'))\n}\n```\n\n### Matcher-Based Path Resolution\n\n```typescript\n// src/utils/hooks/fileChangedWatcher.ts:48-65\nfunction resolveWatchPaths(\n  config?: ReturnType<typeof getHooksConfigFromSnapshot>,\n): string[] {\n  const matchers = (config ?? getHooksConfigFromSnapshot())?.FileChanged ??
