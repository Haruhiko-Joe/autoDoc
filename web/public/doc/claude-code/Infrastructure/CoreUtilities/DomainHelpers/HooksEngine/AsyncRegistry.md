  "content": "# AsyncRegistry\n\n## Overview & Responsibilities\n\nThe AsyncRegistry module lives within **Infrastructure → CoreUtilities → DomainHelpers → HooksEngine** and provides two complementary capabilities:\n\n1. **AsyncHookRegistry** (`src/utils/hooks/AsyncHookRegistry.ts`) — A global registry that tracks shell-based hooks running asynchronously. It manages the full lifecycle of pending hooks: registration, status polling, response collection, finalization, and cleanup.\n\n2. **hookEvents** (`src/utils/hooks/hookEvents.ts`) — A decoupled event broadcasting system for hook lifecycle events. It emits typed events (`HookStarted`, `HookProgress`, `HookResponse`) to a registered handler, with a queue for events emitted before any handler is registered.\n\nTogether, these two files let the rest of the system fire-and-forget async shell hooks and then later poll for their results, while external consumers (e.g., the SDK message stream) receive structured lifecycle events without coupling to the registry internals.\n\n## Key Processes\n\n### Hook Registration Flow\n\n1. A caller invokes `registerPendingAsyncHook()` with a process ID, hook metadata, the `ShellCommand` handle, and the async response (which may specify a custom timeout).\n2. The timeout defaults to **15 seconds** if `asyncResponse.asyncTimeout` is not set (`AsyncHookRegistry.ts:51`).\n3. A progress interval is started via `startHookProgressInterval()`, which polls the shell command's stdout/stderr every second and emits `HookProgress` events when output changes (`hookEvents.ts:124-151`).\n4. The hook is stored in the module-level `pendingHooks` Map keyed by `processId` (`AsyncHookRegistry.ts:69-82`).\n\n### Response Polling Flow (`checkForAsyncHookResponses`)\n\nThis is the core polling function called periodically to harvest completed hooks (`AsyncHookRegistry.ts:113-268`):\n\n1. Snapshots all pending hooks to avoid mutation issues during iteration.\n2. Uses `Promise.allSettled` to process each hook concurrently — isolating failures so one rejection doesn't orphan side-effects from others.\n3. For each hook, determines its state:\n   - **No shell command** → remove from registry\n   - **Killed** → cleanup and remove\n   - **Not completed** → skip (still running)\n   - **Completed but already sent or empty stdout** → remove\n   - **Completed with stdout** → parse output for a JSON response line, mark as delivered, finalize\n4. JSON parsing scans stdout line-by-line for the first JSON object that does **not** contain an `async` key (filtering out the initial async declaration) (`AsyncHookRegistry.ts:192-212`).\n5. After processing, if any `SessionStart` hook completed, the session environment cache is invalidated (`AsyncHookRegistry.ts:257-262`).\n\n### Hook Finalization Flow\n\nThe private `finalizeHook()` function (`AsyncHookRegistry.ts:91-111`):\n\n1. Stops the progress interval\n2. Reads final stdout/stderr from the shell command's `taskOutput`\n3. Cleans up the shell command resources\n4. Emits a `HookResponse` event with the final output, exit code, and outcome (`success`, `error`, or `cancelled`)\n\n### Bulk Finalization (`finalizePendingAsyncHooks`)\n\nUsed during shutdown to ensure no hooks are left dangling (`AsyncHookRegistry.ts:281-301`):\n\n- Completed hooks are finalized with their actual exit code\n- Running hooks are killed and finalized with exit code 1 and `cancelled` outcome\n- The entire registry is cleared afterward\n\n### Event Emission & Filtering (hookEvents)\n\nThe event system uses a `shouldEmit()` gate (`hookEvents.ts:83-91`):\n\n- **Always emitted**: `SessionStart` and `Setup` events — these are low-noise lifecycle events considered backwards-compatible.\n- **Conditionally emitted**: All other hook event types require `allHookEventsEnabled` to be `true` (set via `setAllHookEventsEnabled()` when the SDK's `includeHookEvents` option is active or in `CLAUDE_CODE_REMOTE` mode) **and** the event must be in the `HOOK_EVENTS` allowlist from `coreTypes`.\n\n### Event Queuing\n\nEvents emitted before a handler is registered are buffered in a `pendingEvents` array, capped at **100 entries** (oldest evicted first). When `registerHookEventHandler()` is called, all queued events are immediately flushed to the new handler (`hookEvents.ts:61-70`).\n\n## Function Signatures\n\n### AsyncHookRegistry\n\n#### `registerPendingAsyncHook(params): void`\n\nRegisters a new async hook in the global registry and starts progress broadcasting.\n\n| Parameter | Type | Description |\n|-----------|------|-------------|\n| `processId` | `string` | Unique identifier for this hook process |\n| `hookId` | `string` | Hook definition identifier |\n| `asyncResponse` | `AsyncHookJSONOutput` | Contains `asyncTimeout` (optional, default 15s) |\n| `hookName` | `string` | Human-readable hook name |\n| `hookEvent` | `HookEvent \\| 'StatusLine' \\| 'FileSuggestion'` | The triggering event type |\n| `command` | `string` | Shell command being executed |\n| `shellCommand` | `ShellCommand` | Handle to the running shell process |\n| `toolName` | `string?` | Associated tool name (optional) |\n| `pluginId` | `string?` | Associated plugin ID (optional) |\n\n#### `getPendingAsyncHooks(): PendingAsyncHook[]`\n\nReturns all pending hooks whose responses have **not yet been sent** (filters by `responseAttachmentSent === false`).\n\n#### `checkForAsyncHookResponses(): Promise<Array<{...}>>`\n\nPolls all pending hooks, collects completed responses, and removes finished hooks from the registry. Returns an array of response objects containing `processId`, `response` (parsed JSON output), `hookName`, `hookEvent`, `toolName`, `pluginId`, `stdout`, `stderr`, and `exitCode`.\n\n#### `removeDeliveredAsyncHooks(processIds: string[]): void`\n\nRemoves hooks by process ID, but only if their response has already been sent. Stops progress intervals for removed hooks.\n\n#### `finalizePendingAsyncHooks(): Promise<void>`\n\nGracefully shuts down all remaining hooks — finalizes completed ones and kills/cancels running ones. Clears the entire registry.\n\n#### `clearAllAsyncHooks(): void`\n\nTest utility that stops all progress intervals and clears the registry.\n\n### hookEvents\n\n#### `registerHookEventHandler(handler: HookEventHandler | null): void`\n\nRegisters (or unregisters with `null`) the single event handler. Flushes any queued events on registration.\n\n#### `emitHookStarted(hookId, hookName, hookEvent): void`\n\nEmits a `HookStartedEvent` if the event passes the emission filter.\n\n#### `emitHookProgress(data): void`\n\nEmits a `HookProgressEvent` with current stdout/stderr/output.\n\n#### `startHookProgressInterval(params): () => void`\n\nStarts a periodic timer (default **1000ms**) that polls `getOutput()` and emits progress events only when output changes. Returns a cleanup function to stop the interval. The interval is `unref()`'d so it won't keep the process alive.\n\n#### `emitHookResponse(data): void`\n\nEmits a `HookResponseEvent` with final output, exit code, and outcome. Always logs output to the debug log regardless of emission filtering.\n\n#### `setAllHookEventsEnabled(enabled: boolean): void`\n\nEnables/disables emission of all hook event types beyond the always-emitted `SessionStart` and `Setup`.\n\n#### `clearHookEventState(): void`\n\nResets all event state: nulls the handler, empties the queue, disables all-events mode.\n\n## Type supports only one handler at a time. Registering a new handler replaces the old one; passing `null` unregisters it entirely.\n- **Event queue overflow**: If more than 100 events are emitted before a handler registers, the oldest events are silently dropped (`hookEvents.ts:77-79`).\n- **JSON response parsing**: `checkForAsyncHookResponses` scans stdout line-by-line for the **first** JSON object without an `async` key. Non-JSON lines and JSON parse failures are silently skipped. This means the hook's synchronous response must be a single-line JSON object in stdout.\n- **Session environment invalidation**: Completing a `SessionStart` hook triggers `invalidateSessionEnvCache()`, which forces re-reading of session environment variables. This is a deliberate side-effect at `AsyncHookRegistry.ts:257-262`.\n- **`Promise.allSettled` isolation**: The polling loop uses `allSettled` rather than `all` so that if one hook's processing throws, other hooks' finalization and state mutations are preserved (`AsyncHookRegistry.ts:236-255`).\n- **Interval `unref()`**: Progress intervals call `interval.unref()` so they don't prevent Node.js process exit (`hookEvents.ts:148`).\n- **Response deduplication**: The `responseAttachmentSent` flag prevents the same hook's response from being collected twice across multiple poll cycles.\n- **`emitHookResponse` always logs**: Even when an event is filtered by `shouldEmit()`, the hook's output is always written to the debug log for verbose-mode debugging (`hookEvents.ts:163-169`)."
