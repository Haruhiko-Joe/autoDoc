  "content": "# Hook Registration\n\n## Overview & Responsibilities\n\nThis module provides two functions for registering hooks that are declared in skill or agent frontmatter into the session-scoped hook store. It sits within the **HooksEngine** subsystem of **DomainHelpers → CoreUtilities → Infrastructure**, bridging the gap between declarative hook definitions in frontmatter and the runtime hook execution system.\n\nThe two files serve complementary purposes:\n\n- **`registerSkillHooks`** — Registers skill-defined hooks with support for **one-shot hooks** (`once: true`) that auto-remove after first execution.\n- **`registerFrontmatterHooks`** — A more general registration function for both agent and skill frontmatter hooks, with special handling to **convert `Stop` events to `SubagentStop`** when registering hooks for agents.\n\nBoth delegate to `addSessionHook` (and `removeSessionHook` for one-shot cleanup) from the `sessionHooks` module to persist hooks in app state.\n\n## Key Processes\n\n### registerSkillHooks Flow\n\n1. Iterates over every known `HookEvent` from the `HOOK_EVENTS` constant (PreToolUse, PostToolUse, Stop, SubagentStop, etc.)\n2. For each event, reads the corresponding matchers from the `hooks` settings object\n3. For each matcher/hook combination:\n   - If the hook has `once: true`, creates an `onHookSuccess` callback that calls `removeSessionHook` to auto-remove the hook after its first successful execution (`registerSkillHooks.ts:36-42`)\n   - Calls `addSessionHook` with the event name, matcher pattern, hook definition, the optional success callback, and the skill's root directory (`registerSkillHooks.ts:45-53`)\n4. Logs the total count of registered hooks for debugging\n\n### registerFrontmatterHooks Flow\n\n1. Early-returns if `hooks` is falsy or empty (`registerFrontmatterHooks.ts:25-27`)\n2. Iterates over every known `HookEvent`\n3. **Stop → SubagentStop conversion**: When `isAgent` is `true` and the event is `Stop`, the target event is remapped to `SubagentStop` (`registerFrontmatterHooks.ts:40-45`). This is necessary because when a subagent completes, the hook system fires `SubagentStop` rather than `Stop`.\n4. For each matcher/hook combination, calls `addSessionHook` with the (possibly remapped) target event (`registerFrontmatterHooks.ts:56`)\n5. Logs the total count of registered hooks for debugging\n\n## Function Signatures\n\n### `registerSkillHooks(setAppState, sessionId, hooks, skillName, skillRoot?): void`\n\nRegisters hooks from a skill's frontmatter as session-scoped hooks.\n\n| Parameter | Type | Description |\n|-----------|------|-------------|\n| `setAppState` | `(updater: (prev: AppState) => AppState) => void` | State updater function for modifying app state |\n| `sessionId` | `string` | Current session ID to scope the hooks |\n| `hooks` | `HooksSettings` | Hook definitions from the skill's frontmatter |\n| `skillName` | `string` | Skill name used in debug log messages |\n| `skillRoot` | `string \\| undefined` | Base directory of the skill; passed as `CLAUDE_PLUGIN_ROOT` env var to hook commands |\n\n> Source: `src/utils/hooks/registerSkillHooks.ts:20-26`\n\n### `registerFrontmatterHooks(setAppState, sessionId, hooks, sourceName, isAgent?): void`\n\nRegisters hooks from agent or skill frontmatter into session-scoped hooks.\n\n| Parameter | Type | Default | Description |\n|-----------|------|---------|-------------|\n| `setAppState` | `(updater: (prev: AppState) => AppState) => void` | — | State updater function for modifying app state |\n| `sessionId` | `string` | — | Session or agent ID to scope the hooks |\n| `hooks` | `HooksSettings` | — | Hook definitions from frontmatter |\n| `sourceName` | `string` | — | Human-readable name for logging (e.g., `\"agent 'my-agent'\"`) |\n| `isAgent` | `boolean` | `false` | When `true`, converts `Stop` hooks to `SubagentStop` |\n\n> Source: `src/utils/hooks/registerFrontmatterHooks.ts:18-24`\n\n## Type Definitions\n\n### `HooksSettings`\n\n```typescript\ntype HooksSettings = Partial<Record<HookEvent, HookMatcher[]>>\n```\n\nA partial map from hook event names to arrays of matchers. Defined in `src/schemas/hooks.ts:222`.\n\n### `HookMatcher`\n\nEach matcher contains:\n- **`matcher`** (`string | undefined`) — A pattern string to match against event-specific values (e.g., tool names like `\"Write\"` for `PreToolUse` events)\n- **`hooks`** (`HookCommand[]`) — The list of hook commands to execute when the matcher matches\n\n### `HookEvent`\n\nThe supported hook events include: `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `Notification`, `UserPromptSubmit`, `SessionStart`, `SessionEnd`, `Stop`, `StopFailure`, `SubagentStart`, `SubagentStop`, `PreCompact`, `PostCompact`, `PermissionRequest`, `PermissionDenied`.\n\n## Edge Cases & Caveats\n\n- **One-shot hooks are skill-only**: The `once: true` auto-removal behavior is only supported in `registerSkillHooks`. `registerFrontmatterHooks` does not support one-shot hooks.\n- **Stop → SubagentStop remapping**: When `isAgent` is true, `Stop` hooks are silently converted to `SubagentStop`. This means an agent frontmatter declaring a `Stop` hook will never actually listen for the `Stop` event — it will fire on `SubagentStop` instead. This matches the runtime behavior where `executeStopHooks` fires `SubagentStop` (not `Stop`) when called with an `agentId`.\n- **skillRoot is only passed by registerSkillHooks**: `registerFrontmatterHooks` does not accept or pass a `skillRoot` parameter, so hooks registered through it won't have `CLAUDE_PLUGIN_ROOT` set in their environment.\n- **Empty hooks are handled gracefully**: Both functions short-circuit if no hooks are defined — `registerFrontmatterHooks` does so explicitly at the top, while `registerSkillHooks` naturally skips via the loop conditionals.\n- **Matcher defaults to empty string**: Both functions fall back to `''` when `matcher` is undefined, meaning the hook will match all events of that type."
