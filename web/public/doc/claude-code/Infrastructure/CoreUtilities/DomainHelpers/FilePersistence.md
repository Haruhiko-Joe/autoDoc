  "content": "# FilePersistence\n\n## Overview & Responsibilities\n\nThe FilePersistence module manages **file upload orchestration after each agent turn** in remote session environments. It sits within the **Infrastructure → CoreUtilities → DomainHelpers** layer and is exclusively active in BYOC (Bring Your Own Cloud) remote sessions — normal local CLI users never trigger this code.\n\nIts two files handle complementary concerns:\n- **`filePersistence.ts`** — Orchestration: decides whether persistence is needed, assembles configuration from environment variables and auth tokens, dispatches to the appropriate persistence strategy (BYOC upload or cloud sync), and emits analytics events.\n- **`outputsScanner.ts`** — Detection: reads `CLAUDE_CODE_ENVIRONMENT_KIND` to determine the session type, and performs a filesystem scan to find files modified since the current turn started.\n\n## Key Processes\n\n### End-of-Turn Persistence Flow\n\nThe primary entry point is `executeFilePersistence()`, which wraps `runFilePersistence()` with error handling and a callback-based result delivery pattern.\n\n1. **Environment gate** — `getEnvironmentKind()` reads `CLAUDE_CODE_ENVIRONMENT_KIND`. Only `'byoc'` triggers persistence; `'anthropic_cloud'` and unset values cause an early `null` return (`filePersistence.ts:55-58`).\n2. **Auth gate** — A session ingress auth token must be present (retrieved via `getSessionIngressAuthToken()`). Without it, persistence is silently skipped (`filePersistence.ts:60-63`).\n3. **Session ID gate** — `CLAUDE_CODE_REMOTE_SESSION_ID` must be set. If missing, an error is logged and persistence is skipped (`filePersistence.ts:65-73`).\n4. **Config assembly** — A `FilesApiConfig` object is built from the OAuth token and session ID (`filePersistence.ts:75-78`).\n5. **Outputs directory resolution** — The scan target is `{cwd}/{sessionId}/outputs` (`filePersistence.ts:80`).\n6. **Dispatch to BYOC strategy** — `executeBYOCPersistence()` is called (cloud persistence is stubbed as a TODO for future xattr-based file ID reading).\n7. **Analytics** — `tengu_file_persistence_started` and `tengu_file_persistence_completed` events are logged with duration, success/failure counts, and mode.\n\n### BYOC Upload Flow (`executeBYOCPersistence`)\n\n1. **Scan for modified files** — `findModifiedFiles()` returns files with `mtime >= turnStartTime` (`filePersistence.ts:158`).\n2. **Enforce file count limit** — If more than `FILE_COUNT_LIMIT` files are modified, the entire batch is rejected with a `tengu_file_persistence_limit_exceeded` analytics event (`filePersistence.ts:172-189`).\n3. **Security filter** — Files whose relative path starts with `..` (resolving outside the outputs directory) are silently skipped (`filePersistence.ts:196-203`).\n4. **Parallel upload** — `uploadSessionFiles()` from the Files API service uploads files with `DEFAULT_UPLOAD_CONCURRENCY` (`filePersistence.ts:208-212`).\n5. **Result partitioning** — Upload results are separated into `PersistedFile[]` (success, with `file_id`) and `FailedPersistence[]` (failure, with error message) (`filePersistence.ts:215-230`).\n\n### File Modification Scanning (`findModifiedFiles`)\n\n1. **Recursive directory listing** — Uses `fs.readdir()` with `{ recursive: true, withFileTypes: true }` to get all entries in one call (`outputsScanner.ts:69-72`).\n2. **Symlink exclusion** — Symbolic links are skipped during both the initial `readdir` filter and the subsequent `lstat` check (defense against TOCTOU race conditions) (`outputsScanner.ts:81-83`, `outputsScanner.ts:103`).\n3. **Parallel stat** — All file paths are stat'd concurrently via `Promise.all()` (`outputsScanner.ts:97-111`).\n4. **mtime comparison** — Only files with `mtimeMs >= turnStartTime` are returned (`outputsScanner.ts:116`).\n\n## Function Signatures\n\n### `runFilePersistence(turnStartTime, signal?): Promise<FilesPersistedEventData | null>`\n\nMain orchestrator. Returns persistence results or `null` if persistence is not enabled or there are no files to process.\n\n- **turnStartTime** (`TurnStartTime`) — Timestamp marking the start of the current turn; files modified before this are ignored.\n- **signal** (`AbortSignal`, optional) — Allows cancellation of the persistence operation.\n\n> `filePersistence.ts:51-144`\n\n### `executeFilePersistence(turnStartTime, signal, onResult): Promise<void>`\n\nConvenience wrapper that calls `runFilePersistence` and delivers results via callback. Swallows errors internally.\n\n- **onResult** (`(result: FilesPersistedEventData) => void`) — Called with results if persistence produces any.\n\n> `filePersistence.ts:256-269`\n\n### `isFilePersistenceEnabled(): boolean`\n\nPredicate checking all prerequisites: the `FILE_PERSISTENCE` feature flag, `byoc` environment kind, session access token, and `CLAUDE_CODE_REMOTE_SESSION_ID`.\n\n> `filePersistence.ts:278-287`\n\n### `getEnvironmentKind(): EnvironmentKind | null`\n\nReads `CLAUDE_CODE_ENVIRONMENT_KIND` and returns `'byoc'` or `'anthropic_cloud'` if recognized, otherwise `null`.\n\n> `outputsScanner.ts:25-31`\n\n### `findModifiedFiles(turnStartTime, outputsDir): Promise<string[]>`\n\nScans `outputsDir` recursively and returns absolute paths of regular files modified since `turnStartTime`. Returns `[]` if the directory doesn't exist.\n\n> `outputsScanner.ts:62-126`\n\n## Type Definitions\n\nThe types are imported from `./types.js` (a shared types module):\n\n| Type | Description |\n|------|-------------|\n| `TurnStartTime` | Branded/opaque numeric timestamp marking when a turn began |\n| `PersistedFile` | `{ filename: string, file_id: string }` — successfully uploaded file |\n| `FailedPersistence` | `{ filename: string, error: string }` — file that failed to upload |\n| `FilesPersistedEventData` | `{ files: PersistedFile[], failed: FailedPersistence[] }` — aggregate result |\n| `EnvironmentKind` | `'anthropic_cloud' \\| 'byoc' \\| 'bridge'` (from `teleport/environments.ts`) |\n\n## Constants\n\n| Constant | Description |\n|----------|-------------|\n| `OUTPUTS_SUBDIR` | Subdirectory name under `{cwd}/{sessionId}/` where output files are written |\n| `FILE_COUNT_LIMIT` | Maximum number of files allowed per persistence batch; exceeding this rejects the entire batch |\n| `DEFAULT_UPLOAD_CONCURRENCY` | Number of concurrent upload requests to the Files API |\n\n## Configuration & Environment Variables\n\n| Variable | Required | Description |\n|----------|----------|-------------|\n| `CLAUDE_CODE_ENVIRONMENT_KIND` | Yes | Must be `'byoc'` for persistence to activate. `'anthropic_cloud'` is recognized but cloud persistence is not yet implemented. |\n| `CLAUDE_CODE_REMOTE_SESSION_ID` | Yes | Identifies the remote session; used both for the outputs directory path and the Files API config. |\n| Feature flag `FILE_PERSISTENCE` | Yes | Compile-time feature gate checked by `isFilePersistenceEnabled()`. |\n| Session ingress auth token | Yes | OAuth token retrieved via `getSessionIngressAuthToken()`; used to authenticate Files API uploads. |\n\n## Edge Cases & Caveats\n\n- **Local CLI users are never affected** — All three gates (environment kind, auth token, session ID) must pass. Normal CLI usage has none of these set.\n- **Cloud (1P) persistence is a TODO** — `executeCloudPersistence()` is a stub that returns empty results. The planned approach is xattr-based file ID reading, where rclone handles the actual sync and the scanner just reads file IDs from extended attributes (`filePersistence.ts:246-250`).\n- **Symlink traversal prevention** — Symlinks are excluded at two points: during `readdir` filtering and during `lstat` checks. This prevents symlink-based path traversal attacks (`outputsScanner.ts:81-83`, `outputsScanner.ts:103`).\n- **Path escape prevention** — Files whose relative path resolves outside the outputs directory (starts with `..`) are silently dropped (`filePersistence.ts:198-200`).\n- **TOCTOU handling** — The scanner gracefully handles files that disappear between `readdir` and `stat` (returning `null` for deleted files) and files that become symlinks in the same window (`outputsScanner.ts:100-109`).\n- **Abort signal support** — The abort signal is checked before scanning and before uploading, but does not cancel in-flight uploads.\n- **Batch rejection on count overflow** — If modified files exceed `FILE_COUNT_LIMIT`, *none* are uploaded and a single `FailedPersistence` entry is returned with an explanatory error message.\n- **Node.js compatibility** — The scanner uses `entry.parentPath` (Node 20+) with a fallback to `entry.path` for older versions (`outputsScanner.ts:33-51`).\n- **Analytics events** — Three distinct events track the lifecycle: `tengu_file_persistence_started`, `tengu_file_persistence_completed` (with duration/counts), and `tengu_file_persistence_limit_exceeded`."
