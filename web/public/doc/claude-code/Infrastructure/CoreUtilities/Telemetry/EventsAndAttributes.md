  "content": "# Events and Attributes\n\n## Overview & Responsibilities\n\nThis module provides the shared telemetry foundations used across all observability features in Claude Code. It sits within the **Infrastructure тЖТ CoreUtilities тЖТ Telemetry** layer and serves three distinct purposes:\n\n1. **Structured OTel event emission** (`events.ts`) тАФ a central function for emitting OpenTelemetry log-record events with automatic sequencing, prompt ID tracking, and user-prompt redaction control.\n2. **Skill availability logging** (`skillLoadedEvent.ts`) тАФ records which skills are available at session startup for analytics.\n3. **Telemetry attribute builder** (`telemetryAttributes.ts`) тАФ constructs the standard set of identity and context attributes (user ID, session ID, organization, terminal type) attached to every telemetry signal, with environment-variable-driven cardinality controls.\n\nEvery other telemetry module (traces, metrics, BigQuery export, Perfetto) depends on the attribute builder to produce a consistent set of dimensions. The event emitter is the single gateway for all structured event logging.\n\n## Key Processes\n\n### OTel Event Emission Flow\n\n1. A caller invokes `logOTelEvent(eventName, metadata)` (`src/utils/telemetry/events.ts:21-75`).\n2. The function retrieves the event logger from bootstrap state. If no logger is initialized, it logs a one-time debug warning and drops the event silently.\n3. Events are suppressed entirely when `NODE_ENV === 'test'`.\n4. A base attribute set is built by calling `getTelemetryAttributes()` and merging in:\n   - `event.name` тАФ the raw event name\n   - `event.timestamp` тАФ ISO 8601 timestamp\n   - `event.sequence` тАФ a module-scoped monotonically increasing counter, enabling event ordering within a session\n5. The current `promptId` is fetched from bootstrap state and attached as `prompt.id` (only to events, **not** metrics, to avoid unbounded cardinality).\n6. If the `CLAUDE_CODE_WORKSPACE_HOST_PATHS` environment variable is set (desktop app context), it is split on `|` and attached as `workspace.host_paths`.\n7. Any caller-supplied `metadata` key/value pairs are merged into attributes (undefined values are skipped).\n8. The assembled record is emitted via `eventLogger.emit()` with the body set to `claude_code.<eventName>`.\n\n### User Prompt Redaction\n\nThe `redactIfDisabled()` helper (`src/utils/telemetry/events.ts:17-19`) gates whether user prompt text appears in telemetry. When the `OTEL_LOG_USER_PROMPTS` environment variable is truthy, prompts pass through; otherwise they are replaced with `'<REDACTED>'`. Callers use this to sanitize content before including it in event metadata.\n\n### Skill Loaded Event Logging\n\n1. `logSkillsLoaded(cwd, contextWindowTokens)` is called during session startup (`src/utils/telemetry/skillLoadedEvent.ts:13-39`).\n2. It fetches all available skill commands via `getSkillToolCommands(cwd)` and calculates the character budget for skills using `getCharBudget(contextWindowTokens)`.\n3. For each skill of type `'prompt'`, it emits a `tengu_skill_loaded` analytics event containing:\n   - `_PROTO_skill_name` тАФ the skill's name, routed to a privileged BigQuery column (PII-tagged)\n   - `skill_source` тАФ where the skill originates from\n   - `skill_loaded_from` тАФ how the skill was loaded\n   - `skill_budget` тАФ the character budget allocated\n   - `skill_kind` тАФ optional skill kind, included when present\n\nNote: This function uses the first-party analytics pipeline (`logEvent`) rather than the OTel event logger, and metadata fields use branded type assertions (`AnalyticsMetadata_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS`, `AnalyticsMetadata_I_VERIFIED_THIS_IS_PII_TAGGED`) to enforce data classification at the type level.\n\n### Telemetry Attribute Construction\n\n`getTelemetryAttributes()` (`src/utils/telemetryAttributes.ts:29-71`) builds the standard attribute bag:\n\n1. **User ID** тАФ always included, retrieved via `getOrCreateUserID()` (creates one if absent).\n2. **Session ID** тАФ included by default, controlled by `OTEL_METRICS_INCLUDE_SESSION_ID`.\n3. **App version** тАФ excluded by default, controlled by `OTEL_METRICS_INCLUDE_VERSION`.\n4. **OAuth account data** тАФ only when OAuth authentication is active:\n   - `organization.id` тАФ the organization UUID\n   - `user.email` тАФ the account email\n   - `user.account_uuid` тАФ the raw account UUID (controlled by `OTEL_METRICS_INCLUDE_ACCOUNT_UUID`)\n   - `user.account_id` тАФ a tagged ID form, either from `CLAUDE_CODE_ACCOUNT_TAGGED_ID` env var or computed via `toTaggedId('user', accountUuid)`\n5. **Terminal type** тАФ from the dynamic environment (`envDynamic.terminal`), when available.\n\n## Function Signatures\n\n### `logOTelEvent(eventName: string, metadata?: Record<string, string | undefined>): Promise<void>`\n\nEmits a structured OpenTelemetry log-record event. They intentionally keep `app.version` off by default to limit metric cardinality across deploys.\n\n## Edge Cases & Caveats\n\n- **Event logger not initialized**: If the OTel event logger hasn't been set up (e.g., during early startup), events are silently dropped. A single debug warning is emitted to avoid log spam (`src/utils/telemetry/events.ts:27-33`).\n- **Test environment suppression**: All OTel events are suppressed when `NODE_ENV === 'test'` тАФ even if an event logger exists.\n- **Cardinality discipline**: `prompt.id` and `workspace.host_paths` are deliberately attached only to events, never to metrics, because their high cardinality would explode metric dimensionality and cause issues in the BigQuery metrics pipeline.\n- **PII type branding**: The skill loaded event uses TypeScript branded types (`AnalyticsMetadata_I_VERIFIED_THIS_IS_PII_TAGGED`) as a code-review signal that the developer has verified PII classification тАФ this is a convention, not a runtime check.\n- **Event sequence counter**: The `eventSequence` counter (`src/utils/telemetry/events.ts:8`) is module-scoped and monotonically increasing within a single process. It resets if the process restarts but provides reliable ordering within a session.\n- **OAuth-only fields**: Organization ID, email, and account UUID attributes are only present when the user is authenticated via OAuth тАФ API-key users will not have these dimensions in their telemetry."
