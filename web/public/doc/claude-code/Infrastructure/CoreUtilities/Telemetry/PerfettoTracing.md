  "content": "# Perfetto Tracing\n\n## Overview & Responsibilities\n\nThe Perfetto tracing module (`src/utils/telemetry/perfettoTracing.ts`) is a local performance profiling system that generates traces in **Chrome Trace Event format**, viewable in [ui.perfetto.dev](https://ui.perfetto.dev) or Chrome's `chrome://tracing`. It sits within the **Infrastructure → CoreUtilities → Telemetry** layer alongside OpenTelemetry instrumentation and BigQuery export, but serves a distinct purpose: developer-facing, session-level performance visualization rather than aggregate metrics.\n\nThis feature is **Anthropic-internal only** — the entire module is wrapped in a `feature('PERFETTO_TRACING')` guard that eliminates it via dead-code elimination in external builds.\n\nThe module records four categories of spans:\n- **LLM request spans** — API calls with TTFT/TTLT timing, token statistics, cache hit rates, and retry sub-spans\n- **Tool execution spans** — individual tool invocations with duration and result metadata\n- **User input wait spans** — time spent waiting for user input\n- **Interaction spans** — full user request cycles encompassing API calls and tool executions\n\nIt also manages an **agent hierarchy**, mapping multi-agent (swarm) sessions to Perfetto's process/thread model so parent-child agent relationships render correctly in the trace viewer.\n\n## Key Processes\n\n### Initialization Flow\n\n1. `initializePerfettoTracing()` checks the `CLAUDE_CODE_PERFETTO_TRACE` environment variable (`src/utils/telemetry/perfettoTracing.ts:253-335`)\n2. If the env var is truthy (e.g., `1`), the trace file path defaults to `~/.claude/traces/trace-<sessionId>.json`. If the env var is a path string, that path is used directly\n3. The module registers three exit hooks to ensure the trace file is written:\n   - `registerCleanup()` — async cleanup callback (primary)\n   - `process.on('beforeExit')` — async fallback\n   - `process.on('exit')` — synchronous last-resort via `writePerfettoTraceSync()`\n4. If `CLAUDE_CODE_PERFETTO_WRITE_INTERVAL_S` is set to a positive integer, a periodic write timer is started (with `.unref()` so it won't keep the process alive)\n5. A stale span cleanup interval runs every 60 seconds, evicting spans older than 30 minutes and trimming the event buffer if it exceeds `MAX_EVENTS` (100,000)\n6. Metadata events for the main process/agent are emitted\n\n### Span Lifecycle (Begin/End Pattern)\n\nAll span types follow the same pattern:\n\n1. **Start**: Generate a unique `span_<N>` ID, capture current agent info and timestamp, store a `PendingSpan` in the `pendingSpans` map, and push a `'B'` (begin) event to the event buffer\n2. **End**: Look up the `PendingSpan`, compute duration, merge end-metadata with start-args, push an `'E'` (end) event, and delete the pending entry\n\nIf a span is never explicitly ended, the stale span eviction or `closeOpenSpans()` at shutdown will emit an end event with `evicted: true` or `incomplete: true` respectively.\n\n### LLM Request Span Detail\n\nThe most complex span type. When `endLLMRequestPerfettoSpan()` is called (`src/utils/telemetry/perfettoTracing.ts:468-685`), it:\n\n1. Computes derived metrics:\n   - **ITPS** (input tokens per second): `promptTokens / (ttftMs / 1000)`\n   - **OTPS** (output tokens per second): `outputTokens / (samplingMs / 1000)`\n   - **Cache hit rate**: `(cacheReadTokens / promptTokens) * 100`\n2. Optionally emits a **Request Setup** sub-span covering client creation and retry overhead\n3. Within the setup span, emits individual **retry attempt** sub-spans if `attemptStartTimes` has more than one entry\n4. Emits a **First Token** sub-span (from successful attempt start to TTFT)\n5. Emits a **Sampling** sub-span (from first token to last token)\n6. Closes the parent API Call span\n\nThis nesting produces a rich flame chart: `API Call > [Request Setup > Attempt N] > First Token > Sampling`.\n\n### Event Buffer Management\n\nThe in-memory event buffer is capped at `MAX_EVENTS` (100,000) to prevent unbounded growth in long-running sessions (`src/utils/telemetry/perfettoTracing.ts:232-247`):\n\n1. `evictOldestEvents()` runs every 60 seconds (piggybacks on the stale span cleanup interval)\n2. When `events.length >= MAX_EVENTS`, the oldest half is spliced out\n3. A synthetic `trace_truncated` instant event is inserted so the gap is visible in the Perfetto UI\n4. Metadata events are stored in a separate `metadataEvents` array and are never evicted — Perfetto needs them for track labeling\n\n### Agent Hierarchy & Process/Thread Mapping\n\nPerfetto uses numeric `pid`/`tid` to organize traces into processes and threads. This module maps the agent model onto that (`src/utils/telemetry/perfettoTracing.ts:74-167`):\n\n- The **main session** gets `pid = 1`\n- Each **subagent** gets an incrementing `pid` via `getProcessIdForAgent()`\n- **Thread IDs** are computed as `djb2Hash(agentName)` — deterministic but unique per agent name\n- `registerAgent()` records a new agent and emits three metadata events: `process_name`, `thread_name`, and (if applicable) `parent_agent`\n- `unregisterAgent()` frees the registry entries when an agent completes\n\n## Function Signatures\n\n### `initializePerfettoTracing(): void`\nInitializes the tracer. Call once early in the application lifecycle. Reads `CLAUDE_CODE_PERFETTO_TRACE` and `CLAUDE_CODE_PERFETTO_WRITE_INTERVAL_S` from the environment.\n\n### `isPerfettoTracingEnabled(): boolean`\nReturns whether tracing is currently active.\n\n### `registerAgent(agentId: string, agentName: string, parentAgentId?: string): void`\nRegisters a subagent in the trace, assigning it a process ID and emitting metadata events.\n\n### `unregisterAgent(agentId: string): void`\nRemoves an agent from the registry. Call when an agent completes or is aborted.\n\n### `startLLMRequestPerfettoSpan(args: { model, promptTokens?, messageId?, isSpeculative?, querySource? }): string`\nBegins an API call span. Returns a span ID to pass to `endLLMRequestPerfettoSpan`.\n\n### `endLLMRequestPerfettoSpan(spanId: string, metadata: { ttftMs?, ttltMs?, promptTokens?, outputTokens?, cacheReadTokens?, cacheCreationTokens?, messageId?, success?, error?, requestSetupMs?, attemptStartTimes? }): void`\nCloses an API call span, emitting sub-spans for request setup, retries, TTFT, and sampling phases.\n\n### `startToolPerfettoSpan(toolName: string, args?: Record<string, unknown>): string`\nBegins a tool execution span. Returns a span ID.\n\n### `endToolPerfettoSpan(spanId: string, metadata?: { success?, error?, resultTokens? }): void`\nCloses a tool execution span.\n\n### `startUserInputPerfettoSpan(context?: string): string`\nBegins a user-input-wait span. Returns a span ID.\n\n### `endUserInputPerfettoSpan(spanId: string, metadata?: { decision?, source? }): void`\nCloses a user-input-wait span.\n\n### `startInteractionPerfettoSpan(userPrompt?: string): string`\nBegins an interaction span wrapping a full user request cycle. Returns a span ID.\n\n### `endInteractionPerfettoSpan(spanId: string): void`\nCloses an interaction span.\n\n### `emitPerfettoInstant(name: string, category: string, args?: Record<string, unknown>): void`\nEmits a point-in-time marker event.\n\n### `emitPerfettoCounter(name: string, values: Record<string, number>): void`\nEmits a counter event for tracking numeric metrics over time.\n\n## Type Definitions\n\n### `TraceEventPhase`\nUnion of Chrome Trace Event phase characters: `'B'` (begin), `'E'` (end), `'X'` (complete), `'i'` (instant), `'C'` (counter), `'b'`/`'n'`/`'e'` (async), `'M'` (metadata).\n\n### `TraceEvent`\nThe core event structure written to the trace file:\n\n| Field | Type | Description |\n|-------|------|-------------|\n| `name` | `string` | Event name displayed in the trace viewer |\n| `cat` | `string` | Category (e.g., `api`, `tool`, `user_input`, `interaction`) |\n| `ph` | `TraceEventPhase` | Event phase |\n| `ts` | `number` | Timestamp in microseconds relative to trace start |\n| `pid` | `number` | Process ID (1 for main, incremented for subagents) |\n| `tid` | `number` | Thread ID (DJB2 hash of agent name) |\n| `dur?` | `number` | Duration in microseconds (for `'X'` events) |\n| `args?` | `Record<string, unknown>` | Arbitrary metadata |\n\n### `AgentInfo`\nInternal structure tracking an agent's identity and Perfetto coordinates: `agentId`, `agentName`, `parentAgentId?`, `processId`, `threadId`.\n\n### `PendingSpan`\nBookkeeping for open spans awaiting their end event: `name`, `category`, `startTime`, `agentInfo`, `args`.\n\n## Configuration\n\n| Environment Variable | Required | Default | Description |\n|---------------------|----------|---------|-------------|\n| `CLAUDE_CODE_PERFETTO_TRACE` | Yes (to enable) | unset | Set to `1` for default path (`~/.claude/traces/trace-<sessionId>.json`) or to a file path for a custom location |\n| `CLAUDE_CODE_PERFETTO_WRITE_INTERVAL_S` | No | write only on exit | Positive integer — seconds between periodic full-trace writes to disk |\n\n## Edge Cases & Caveats\n\n- **Double-write prevention**: The `traceWritten` flag ensures the final trace is written exactly once, even though three exit handlers are registered (cleanup registry, `beforeExit`, `exit`). The synchronous `exit` handler is the last resort when async handlers couldn't complete.\n- **Stale span eviction**: Spans open longer than 30 minutes are auto-closed with an `evicted: true` flag. This prevents memory leaks from spans that were never explicitly ended (e.g., due to crashes or aborted agents).\n- **Event buffer cap**: At 100,000 events (~30 MB), the oldest half is dropped with a `trace_truncated` marker. This prevents unbounded memory growth in long-lived cron-driven sessions.\n- **Metadata events are never evicted**: They live in a separate `metadataEvents` array because Perfetto UI needs them to label process/thread tracks correctly.\n- **Timer `.unref()`**: Both the periodic write and stale span cleanup timers call `.unref()` so they don't prevent Node.js from exiting naturally.\n- **All public functions are no-ops when disabled**: Every exported function checks `isEnabled` at entry and returns early (returning `''` for span IDs), so callers don't need guard checks.\n- **External build elimination**: The entire initialization body is wrapped in `feature('PERFETTO_TRACING')`, which the build system replaces with `false` in external builds, allowing the bundler to tree-shake the dead code."
