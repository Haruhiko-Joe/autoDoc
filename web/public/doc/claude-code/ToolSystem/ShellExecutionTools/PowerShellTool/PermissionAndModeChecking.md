  "content": "# Permission and Mode Checking\n\n## Overview & Responsibilities\n\nThis module implements the PowerShell-specific permission checking subsystem, responsible for deciding whether a given PowerShell command should be **allowed**, **denied**, or **require user approval** before execution. It sits within the **PowerShellTool** (under **ShellExecutionTools → ToolSystem**), mirroring the BashTool's permission architecture but adapted for case-insensitive cmdlet matching, PowerShell alias resolution, and PowerShell-specific security concerns.\n\nThe module is split across two files:\n\n- **`powershellPermissions.ts`** — The main orchestrator (`powershellToolHasPermission`) that parses commands, applies user-configured permission rules (deny/ask/allow with wildcard pattern matching), delegates to security analysis and path validation, and generates permission request messages for the UI dialog.\n- **`modeValidation.ts`** — The `acceptEdits` permission mode handler that auto-allows filesystem-modifying cmdlets (like `Set-Content`, `Remove-Item`) when the user has opted into edit mode, with extensive security guards against code smuggling.\n\n## Key Processes\n\n### Main Permission Flow (`powershellToolHasPermission`)\n\nThe orchestrator at `powershellPermissions.ts:639-1648` implements a multi-stage pipeline with a **collect-then-reduce** architecture:\n\n1. **Empty command check** — Empty commands are auto-allowed (`powershellPermissions.ts:647-656`)\n\n2. **Parse the command** — Calls `parsePowerShellCommand()` once and threads the AST through all sub-functions (`powershellPermissions.ts:659`)\n\n3. **Pre-parse deny/ask rules** — Deny and ask rules are checked BEFORE parse validity, ensuring explicit rules work even when `pwsh` is unavailable (`powershellPermissions.ts:661-711`). UNC path detection also runs here (`powershellPermissions.ts:717-723`)\n\n4. **Parse-failed fallback** — When parsing fails (pwsh unavailable, timeout, bad syntax), a conservative fallback splits the raw command on PowerShell separators (`;|\\n{}()&`) and checks each fragment against deny rules. Also normalizes assignment prefixes (`$x = iex ...`) and invocation operators (`& 'cmd'`) before matching (`powershellPermissions.ts:764-873`)\n\n5. **Collect-then-reduce phase** (post-parse) — All checks push decisions into a `decisions[]` array; a single reduce applies **deny > ask > allow > passthrough** precedence (`powershellPermissions.ts:900-1368`). This structurally prevents an earlier `ask` from masking a later `deny`. Checks include:\n   - Security analysis (`powershellCommandIsSafe`) — detects subexpressions, script blocks, encoded commands, download cradles\n   - `using` statements and `#Requires` directives that load external code\n   - Non-filesystem provider paths (`env:`, `HKLM:`, `function:`) and UNC paths in resolved args\n   - Per-sub-command deny/ask rules against both raw text and canonical (alias-resolved) forms\n   - Compound cd+git guard — prevents bare repository attacks when `Set-Location` is paired with `git`\n   - Bare git repo detection — warns when cwd looks like a bare repo\n   - Git-internal path write guard — detects writes to `hooks/`, `refs/`, `.git/` paths\n   - Archive extractor + git TOCTOU guard\n   - Path constraint validation (delegates to `pathValidation.ts`)\n   - Read-only command allowlist check\n   - File redirection detection\n   - `acceptEdits` mode check (delegates to `modeValidation.ts`)\n\n6. **Per-sub-command approval** — Commands not resolved by the reduce are checked individually through `powershellToolCheckPermission`, the read-only allowlist, and per-statement `checkPermissionMode`. A fail-closed gate ensures statements with no `CommandAst` sub-commands (e.g., bare `$env:SECRET`) are never silently auto-allowed (`powershellPermissions.ts:1370-1626`)\n\n7. **Suggestion generation** — Sub-commands needing approval produce `PermissionUpdate` suggestions for the permission dialog (`powershellPermissions.ts:1628-1648`)\n\n### Permission Rule Matching (`filterRulesByContentsMatchingInput`)\n\nRule matching (`powershellPermissions.ts:170-333`) supports three rule types:\n\n- **Exact** — Case-insensitive full command match\n- **Prefix** — Case-insensitive command prefix (e.g., `Get-Process:*` matches `Get-Process -Name foo`)\n- **Wildcard** — Pattern matching via `matchWildcardPattern` (case-insensitive)\n\nAll matching is **alias-aware**: both the input command and the rule's command name are resolved to their canonical form via `resolveToCanonical()`. This means a deny rule on `rm` also blocks `Remove-Item`, `del`, `ri`, etc., and vice versa. The module-prefix stripping behavior differs by rule behavior — deny/ask rules strip module prefixes (fail-safe over-match), while allow rules do not (fail-safe under-match) (`powershellPermissions.ts:189-194`).\n\nWhitespace is normalized to a single space between cmdlet name and arguments to prevent bypass via tab characters (`powershellPermissions.ts:209-215`).\n\n### acceptEdits Mode Validation (`checkPermissionMode`)\n\nWhen the permission mode is `acceptEdits`, `modeValidation.ts:132-404` auto-allows commands where **every** cmdlet is a known filesystem-modifying operation:\n\n**Allowed cmdlets** (`modeValidation.ts:33-38`):\n- `Set-Content`, `Add-Content`, `Remove-Item`, `Clear-Content`\n- Plus their aliases (e.g., `sc`, `ac`, `rm`, `del`) via `resolveToCanonical()`\n\n**Security gates** (any failure causes fallthrough to `ask`):\n1. Parse must be valid\n2. No subexpressions, script blocks, member invocations, splatting, assignments, stop-parsing tokens, or expandable strings (`modeValidation.ts:163-179`)\n3. No empty pipeline segments\n4. **Compound cwd desync guard** — rejects when a cwd-changing cmdlet (`Set-Location`, `Push-Location`, etc.) coexists with a write cmdlet, since path validation uses stale cwd (`modeValidation.ts:192-242`)\n5. **Symlink compound guard** — rejects when `New-Item -ItemType SymbolicLink/Junction/HardLink` coexists with any other command (`modeValidation.ts:235-241`)\n6. No non-`CommandAst` pipeline elements (blocks expression sources like `'/etc/passwd' | Remove-Item`)\n7. No `application` nameType (blocks path-based names like `scripts\\Remove-Item`)\n8. Element type whitelist — only `StringConstant` and `Parameter` argument types allowed; rejects `Variable`, `Other`, etc.\n9. Colon-bound expression metachar check — rejects `-Path:(1 > /tmp/x)`\n10. `argLeaksValue` gate — rejects unvalidatable argument types\n\n### Symlink Detection (`isSymlinkCreatingCommand`)\n\n`modeValidation.ts:82-117` detects `New-Item` commands that create filesystem links. It handles:\n- Parameter abbreviation (`-it`, `-ite`, ..., `-itemtype`; `-ty`, `-typ`, `-type`)\n- Unicode dash prefixes (en-dash `–`, em-dash `—`, horizontal bar `―`) and forward-slash `/` as parameter markers\n- Colon-bound values (`-it:SymbolicLink`)\n- Backtick escapes (`-Item\\`Type`, `Sym\\`bolicLink`)\n- Surrounding quotes (`-it:'SymbolicLink'`, `-it:\"Junction\"`)\n\nDetected link types: `SymbolicLink`, `Junction`, `HardLink`.\n\n## Function Signatures\n\n### `powershellToolHasPermission(input, context): Promise<PermissionResult>`\n\nMain entry point. Orchestrates the full permission flow.\n\n- **input.command** (`string`) — The PowerShell command to check\n- **input.timeout** (`number`, optional) — Command timeout\n- **context** (`ToolUseContext`) — Provides access to app state and permission context\n- **Returns**: `PermissionResult` with `behavior` of `'allow'`, `'deny'`, `'ask'`, or `'passthrough'`\n\n> Source: `powershellPermissions.ts:639-1648`\n\n### `powershellToolCheckPermission(input, toolPermissionContext): PermissionResult`\n\nChecks permission rules (exact then prefix/wildcard) without security analysis. Used by the main orchestrator for per-sub-command rule checks.\n\n- Priority order: exact deny → exact ask → prefix deny → prefix ask → exact allow → prefix allow → passthrough\n\n> Source: `powershellPermissions.ts:435-514`\n\n### `powershellToolCheckExactMatchPermission(input, toolPermissionContext): PermissionResult`\n\nChecks only exact-match rules. Returns the first matching deny, ask, or allow rule; passthrough if none match.\n\n> Source: `powershellPermissions.ts:385-430`\n\n### `checkPermissionMode(input, parsed, toolPermissionContext): PermissionResult`\n\nChecks if the current permission mode auto-allows the command. Only acts in `acceptEdits` mode; returns `passthrough` for all other modes.\n\n> Source: `modeValidation.ts:132-404`\n\n### `isSymlinkCreatingCommand(cmd): boolean`\n\nDetects `New-Item` commands creating filesystem links.\n\n- **cmd.name** (`string`) — Command name (resolved via `resolveToCanonical`)\n- **cmd.args** (`string[]`) — Command arguments\n\n> Source: `modeValidation.ts:82-117`\n\n### `powershellPermissionRule(permissionRule): ShellPermissionRule`\n\nParses a permission rule string into a structured object (delegates to `parsePermissionRule`).\n\n> Source: `powershellPermissions.ts:132-136`\n\n## Type Definitions\n\n### `PowerShellInput`\n\n| Field | Type | Description |\n|-------|------|-------------|\n| command | `string` | The PowerShell command string |\n| timeout | `number?` | Optional execution timeout |\n\n### `SubCommandInfo`\n\n| Field | Type | Description |\n|-------|------|-------------|\n| text | `string` | Raw text of the sub-command |\n| element | `ParsedCommandElement` | AST element with name, args, elementTypes, nameType, redirections |\n| statement | `ParsedPowerShellCommand['statements'][number] \\| null` | Parent statement (for nested command tracking) |\n| isSafeOutput | `boolean` | Whether this is a safe output cmdlet (Out-Null, etc.) with zero args |\n\n### `PermissionResult` (imported)\n\nThe return type for all permission checks. Key fields:\n- `behavior`: `'allow'` | `'deny'` | `'ask'` | `'passthrough'`\n- `message`: Human-readable explanation\n- `updatedInput`: Present when `behavior === 'allow'`\n- `decisionReason`: `{ type: 'rule', rule } | { type: 'mode', mode } | { type: 'other', reason }`\n- `suggestions`: `PermissionUpdate[]` — Rule suggestions for the permission dialog\n\n## Configuration & Defaults\n\n### `ACCEPT_EDITS_ALLOWED_CMDLETS` (`modeValidation.ts:33-38`)\nThe set of cmdlets auto-allowed in `acceptEdits` mode: `set-content`, `add-content`, `remove-item`, `clear-content`. Tier 3 cmdlets (`new-item`, `copy-item`, `move-item`) are intentionally excluded due to complex parameter binding.\n\n### `GIT_SAFETY_WRITE_CMDLETS` (`powershellPermissions.ts:70-84`)\nCmdlets that can create files at caller-specified paths, checked against git-internal path patterns: `new-item`, `set-content`, `add-content`, `out-file`, `copy-item`, `move-item`, `rename-item`, `expand-archive`, `invoke-webrequest`, `invoke-restmethod`, `tee-object`, `export-csv`, `export-clixml`.\n\n### `GIT_SAFETY_ARCHIVE_EXTRACTORS` (`powershellPermissions.ts:96-112`)\nExternal archive tools whose contents are opaque: `tar`, `bsdtar`, `unzip`, `7z`, `7za`, `gzip`, `gunzip`, `expand-archive` (with `.exe` variants).\n\n### `LINK_ITEM_TYPES` (`modeValidation.ts:56`)\nNew-Item types that create filesystem links: `symboliclink`, `junction`, `hardlink`.\n\n## Edge Cases & Caveats\n\n- **Parse-failed degradation**: When `pwsh` is unavailable or the command exceeds `MAX_COMMAND_LENGTH`, the module degrades to raw-string fragment scanning. This is conservative — it may false-positive deny on fragments inside string literals, but never false-negative (fail-safe).\n\n- **Module-qualified names**: `Microsoft.PowerShell.Management\\Remove-Item` has its module prefix stripped for matching. For deny/ask rules this is fail-safe (over-match); for allow rules the prefix is kept (under-match prevents cross-module allow bypass).\n\n- **Exact command suggestions are skipped** for multi-line commands and commands containing literal `*` (`powershellPermissions.ts:151-153`). Multi-line commands don't survive normalization; literal `*` would re-parse as a wildcard rule.\n\n- **Safe output cmdlets** (`Out-Null`, `Format-Table`, etc.) are checked against deny rules but otherwise filtered from the approval list — they inherit permission from the preceding pipeline command. However, they are only treated as safe when they have zero arguments (`powershellPermissions.ts:582-585`).\n\n- **cd-to-CWD filtering**: Standalone `Set-Location .` (resolving to current directory) is filtered from the approval list as a no-op. But in compound commands, ANY cd-family cmdlet triggers the compound guard regardless.\n\n- **Collect-then-reduce vs early-return**: The module explicitly avoids early-return for ask decisions because an earlier `ask` could mask a later `deny` on a different sub-command. The `decisions[]` array with reduce ensures deny always wins.\n\n- **Fail-closed statement gate** (`powershellPermissions.ts:1593-1597`): Statements that produce no `CommandAst` sub-commands (e.g., bare `$env:SECRET`) or whose sub-commands were all filtered as safe-output are caught by a post-loop sweep that pushes them into the approval list. This prevents silent auto-allow of invisible non-command content."
