# doc-drill

You have access to structured, hierarchical documentation generated by autoDoc, served over HTTP by the **`autodoc` MCP server**. The docs are organized as a tree: a top-level graph per project, sub-graphs nested inside modules, markdown leaf pages at the bottom, and interaction flows for classic end-to-end cases.

**Do not try to dump everything at once.** Drill down one layer at a time, guided by relevance to the task.

## Transport

All navigation and mutation happens through MCP tools exposed by the HTTP server named `autodoc`. autoDoc writes the project MCP configuration when it installs this skill: Claude Code reads `.mcp.json`, while Codex reads `.codex/config.toml`. The server is expected to be available at the configured `/mcp` URL. You never read doc files directly from disk ‚Äî always go through these tools.

## Documentation model

Each project looks like this on the server:

```
{project}/
‚îú‚îÄ‚îÄ top.json              # project overview + top-level modules (a TopGraph)
‚îú‚îÄ‚îÄ flows.json            # typical cross-module interaction flows
‚îú‚îÄ‚îÄ {Module}/
‚îÇ   ‚îú‚îÄ‚îÄ {Module}.json     # a sub-Graph: description, codeScope, child nodes
‚îÇ   ‚îú‚îÄ‚îÄ {Leaf}.md         # leaf page: detailed doc for one component
‚îÇ   ‚îî‚îÄ‚îÄ {SubModule}/
‚îÇ       ‚îú‚îÄ‚îÄ {SubModule}.json
‚îÇ       ‚îî‚îÄ‚îÄ ...
```

A graph's child node either points to a deeper sub-graph (`child.type == "graph"`) or to a markdown leaf page (`child.type == "page"`). Each graph also carries:
- `codeScope` ‚Äî the exact source files/dirs this module documents

Edges between sibling nodes carry relationship types: `calls`, `depends`, `data-flow`, `event`, `extends`, `composes`.

## Read tools (start here)

### Structured navigation

| Tool | When to use |
|---|---|
| `list_projects` | See which projects are documented on this server |
| `get_top(project)` | Get the project overview and top-level module list ‚Äî **always call this first** |
| `get_flows(project)` | Read typical cross-module interaction flows. Use this for classic cases and "how do modules work together?" questions |
| `get_graph(project, nodeId)` | Drill into a module. Returns `codeScope`, `nodes`. `nodeId` is slash-separated, e.g. `MobileApps/IOSApp` |
| `get_page(project, nodeId, ref)` | Read a leaf markdown page. Returns `{ content }` |
| `search_nodes(project, query)` | Keyword-search names + descriptions across every level. Returns full nodeId path so you can jump straight to the right depth |

### Batch / source access

| Tool | When to use |
|---|---|
| `list_source_files(project, patterns)` | List repo source files matching regex patterns. Pass `['.*']` for all |
| `read_source_files(project, requests)` | Read source code files with optional line-range slicing. Each request is `{ path, start, end }` |
| `list_docs(project, patterns)` | List doc nodeIds matching regex patterns. Pass `['.*']` for all |
| `read_docs(project, paths)` | Batch-read doc files by nodeId (returns raw content ‚Äî prefer `get_top`/`get_graph`/`get_page` for structured reads) |

## Write tools (only when the user asks you to maintain docs)

Write tools update the documentation working tree only. A human reviews and commits the accumulated documentation changes from autoDoc's web UI.

| Tool | Purpose |
|---|---|
| `update_top` | Patch `top.json`'s `description` and/or full `nodes` array |
| `update_graph_meta` | Patch a subgraph's `description` and/or `codeScope` |
| `create_node` | Append a new child node to a parent graph (page or sub-graph). For pages, optionally seed `initialContent` |
| `update_node` | Patch an existing child node's `name` / `description` / `codeScope` / `edges` |
| `delete_node` | Remove a child node. Pages delete the `.md`; graph children are removed recursively |
| `patch_page` | Apply targeted string-match edits to a page without rewriting the full content. Each edit is `{ old_text, new_text }`. **Prefer this over `update_page` for small changes** ‚Äî cheaper and less error-prone |
| `update_page` | Overwrite a leaf markdown page's full content. Use only when the page needs major restructuring |

## Progressive-disclosure workflow

Follow this discipline every time you need codebase context:

### 1. Orient ‚Äî `get_top`
Read the module list, their descriptions, and edges to understand the high-level architecture. **Stop here if the task only needs architectural understanding.**

### 2. Classic cases ‚Äî `get_flows`
For cross-module behavior, user journeys, request lifecycles, sync jobs, or "how does this work end to end?" questions, read flows before drilling. Use the participants, `docPath`, and `codeRef` fields as navigation hints. If `get_flows` reports that flows have not been generated yet, continue with `get_top` / `get_graph` / `get_page` and source reads.

### 3. Locate ‚Äî `get_graph`
Pick the 1‚Äì2 most relevant modules based on descriptions and `codeScope`. Drill into them. Read child node descriptions and edges. **Stop here if you can now identify the right component.**

### 4. Focus ‚Äî `get_page`
Once you've identified the specific component, read its `.md` page. You now have detailed implementation docs: file-by-file breakdowns, data flows, key APIs.

### 5. Search ‚Äî `search_nodes`
If you can't tell which module to drill into, search by keyword. The results tell you the matching node's path so you can jump straight to the right depth.

## Writing workflow

When the user asks you to change the documentation:

1. **Read first** ‚Äî call `get_top`, `get_graph`, or `get_page` to understand the current state
2. **Choose the right tool**:
   - Small page edits (fixing a paragraph, updating an API name): use `patch_page` with `[{ old_text, new_text }]`
   - Major page rewrites: use `update_page` with full content
   - Graph metadata: use `update_graph_meta` or `update_node`
   - Structural changes: use `create_node` / `delete_node`
3. **Write** ‚Äî each mutation updates the working tree; the user will commit later in the web UI

For new content, prefer `create_node` with `initialContent` for pages, or `create_node` with `child.type="graph"` for a new sub-module placeholder.

## Key principles

- **Lazy loading**: Never read leaf pages until you've confirmed they're relevant via the graph hierarchy. Each page can be 5‚Äì15 KB ‚Äî loading everything wastes context
- **Flow-guided tracing**: For end-to-end behavior, start from `get_flows`, then drill only into the modules and pages named by the relevant case
- **Edge-guided navigation**: The `edges` array tells you which siblings interact. Tracing a data flow or call chain? Follow the edges rather than reading every child
- **codeScope is ground truth**: When mapping docs back to source files, read `codeScope`. It lists the exact files/dirs a node documents
- **Breadth before depth**: At each level, scan all node descriptions first before drilling. Don't commit to the first plausible match

## Output conventions

When you cite documentation findings in your answers:
- Use the module path as the citation: `Core/SessionEngine/TurnLifecycle`
- For end-to-end behavior, mention the flow title and the participant path(s) you used
- Mention source files from `codeScope` when the user needs to edit code
- If a question spans multiple modules, trace the edges between them to explain the connection
