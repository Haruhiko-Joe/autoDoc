# doc-drill

You have access to structured, hierarchical documentation generated by autoDoc, served over HTTP by the **`autodoc` MCP server**. The docs are organized as a tree: a top-level graph per project, sub-graphs nested inside modules, and markdown leaf pages at the bottom.

**Do not try to dump everything at once.** Drill down one layer at a time, guided by relevance to the task.

## Transport

All navigation and mutation happens through MCP tools exposed by the server named `autodoc`. The user configures the endpoint in their `.mcp.json` (typically `http://<host>:3200/mcp`). You never read doc files directly from disk — always go through these tools.

## Documentation model

Each project looks like this on the server:

```
{project}/
├── top.json              # project overview + top-level modules (a TopGraph)
├── {Module}/
│   ├── {Module}.json     # a sub-Graph: description, codeScope, child nodes
│   ├── {Leaf}.md         # leaf page: detailed doc for one component
│   └── {SubModule}/
│       ├── {SubModule}.json
│       └── ...
```

A graph's child node either points to a deeper sub-graph (`child.type == "graph"`) or to a markdown leaf page (`child.type == "page"`). Each graph also carries:
- `version` — integer, bumped on every write; pass it back as `baseVersion` for any graph-level mutation
- `pageVersions` — per-page version map; use `pageVersions[ref]` as `baseVersion` when editing a leaf page
- `codeScope` — the exact source files/dirs this module documents

Edges between sibling nodes carry relationship types: `calls`, `depends`, `data-flow`, `event`, `extends`, `composes`.

## Read tools (start here)

| Tool | When to use |
|---|---|
| `list_projects` | See which projects are documented on this server |
| `get_top(project)` | Get the project overview and top-level module list — **always call this first** |
| `get_graph(project, nodeId)` | Drill into a module. `nodeId` is a slash-separated path like `MobileApps/IOSApp` |
| `get_page(project, nodeId, ref)` | Read a leaf markdown page. `nodeId` = parent graph path, `ref` = child's `ref` |
| `search_nodes(project, query)` | Keyword-search names + descriptions across every level when you don't know which module to pick |
| `list_history(project, relPath)` | List historical versions of a file (e.g. `top.json`, `MobileApps/MobileApps.json`, `MobileApps/IOSApp/VoiceSystem.md`) |
| `get_history(project, relPath, version)` | Read a specific historical version's content |

## Write tools (only when the user asks you to maintain docs)

Every write tool requires a `baseVersion`. Get it by **reading the file first**, then pass back the `version` (or `pageVersions[ref]`) you just saw. If someone else wrote in between, the server returns a VersionMismatch error — re-read and retry.

| Tool | Purpose |
|---|---|
| `update_top` | Patch `top.json`'s `description` and/or full `nodes` array |
| `update_graph_meta` | Patch a subgraph's `description` and/or `codeScope` |
| `create_node` | Append a new child node to a parent graph (page or sub-graph). For pages, optionally seed `initialContent` |
| `update_node` | Patch an existing child node's `name` / `description` / `codeScope` / `edges` |
| `delete_node` | Remove a child node. Pages delete the `.md`; graph children are removed recursively |
| `update_page` | Overwrite a leaf markdown page's full content. `baseVersion` comes from `pageVersions[ref]`, not the graph's own `version` |
| `revert` | Make a past snapshot the new current version (creates version N+1 with old content; nothing is erased) |

## Progressive-disclosure workflow

Follow this discipline every time you need codebase context:

### 1. Orient — `get_top`
Read the module list, their descriptions, and edges to understand the high-level architecture. **Stop here if the task only needs architectural understanding.**

### 2. Locate — `get_graph`
Pick the 1–2 most relevant modules based on descriptions and `codeScope`. Drill into them. Read child node descriptions and edges. **Stop here if you can now identify the right component.**

### 3. Focus — `get_page`
Once you've identified the specific component, read its `.md` page. You now have detailed implementation docs: file-by-file breakdowns, data flows, key APIs.

### 4. Search — `search_nodes`
If you can't tell which module to drill into, search by keyword. The results tell you the matching node's path so you can jump straight to the right depth.

## Writing workflow

When the user asks you to change the documentation:

1. **Read first** — always call the matching read tool to get the current object and its `version` / `pageVersions[ref]`
2. **Compute the new value** locally (don't ask the server to do partial merges beyond the patch fields the tool already supports)
3. **Write** with the `baseVersion` you just read
4. **On VersionMismatch**, re-read and retry (the error message tells you the server's current version)

For new content, prefer `create_node` with `initialContent` for pages, or `create_node` with `child.type="graph"` for a new sub-module placeholder.

## Key principles

- **Lazy loading**: Never read leaf pages until you've confirmed they're relevant via the graph hierarchy. Each page can be 5–15 KB — loading everything wastes context
- **Edge-guided navigation**: The `edges` array tells you which siblings interact. Tracing a data flow or call chain? Follow the edges rather than reading every child
- **codeScope is ground truth**: When mapping docs back to source files, read `codeScope`. It lists the exact files/dirs a node documents
- **Breadth before depth**: At each level, scan all node descriptions first before drilling. Don't commit to the first plausible match

## Output conventions

When you cite documentation findings in your answers:
- Use the module path as the citation: `Core/SessionEngine/TurnLifecycle`
- Mention source files from `codeScope` when the user needs to edit code
- If a question spans multiple modules, trace the edges between them to explain the connection
